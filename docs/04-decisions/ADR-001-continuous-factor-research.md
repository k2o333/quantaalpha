# ADR-001: QuantaAlpha 持续因子研究系统演进

Status: draft
Owner: 
Created: 2026-03-15
Outcome: pending

## 1. Context & Problem Statement (背景与问题)

当前 QuantaAlpha 系统具备单次因子挖掘与回测的主链路，但在“持续研究”方面存在以下局限性：
- **生成单一化**：挖掘阶段过于依赖单模型或单配置，缺乏多模型路由、不同视角的交叉验证以及输出错误时的独立修复链路。
- **验证单薄**：回测仅针对单套时间区间（如单一的 train/valid/test），无法有效评估因子在多时间跨度、多周期下的长期稳定性。
- **知识沉淀不足**：因子库仅作为回测结果的简单归档，缺乏多维度状态（如依赖数据字段、稳定性画像、演化来源、失效信息等），难以指导未来的因子挖掘。
- **数据维度感知弱**：引入新数据维度（如基本面、资金流等）时，LLM 缺乏对可用数据资产、字段频率和对齐逻辑的全局感知。
- **缺乏持续运转的闭环**：当新数据到达或因子长期未回测时，系统无法进行自动扫描、筛选、复验和维护。

## 2. Decision Drivers (决策驱动力)

- 将系统从“单次因子挖掘与回测工具”升级为支持 24 小时运转的“持续因子研究系统”。
- 解决模型调用成本控制与生成多维度的平衡。
- 提升因子挖掘的长期有效性与稳定性。
- 将因子的产出转变为可复用、可变异、可检索的研究知识资产。
- 采用分阶段、小步闭环的演进方式，在现有系统框架上增加能力层，而非推倒重来。

## 3. Decision (决策)

我们决定在现有主链之上增加四层核心能力，形成一套松耦合的持续研究闭环：

### 3.1 引入多模型研究层 (Multi-Model Research)
- **拆分模型调度职责**：将任务按复杂度智能路由到不同模型（大推理模型处理假设生成，中小模型处理初筛，Coding 模型专注结构修复）。
- **平台并发与解耦**：引入 `ProviderPool` 支持平台轮询以分摊压力，并支持同任务下多模型并发研讨（Fanout机制）以增加多样性。

### 3.2 建立多周期验证引擎 (Multi-Period Validation Engine)
- 在原有单区间回测的基础上，封装多周期回测层。
- 评估因子的跨周期稳定性，以多个评价周期（Period）的表现综合评分考察因子，取代单一的测试集结果。

### 3.3 升级为因子知识库 (Factor Knowledge Library)
- 重构现有因子存储 Schema，记录因子的高级元数据，如：数据字段要求、跨周期稳定性、使用建议、以及因子演化族谱（父代来源、模型来源等）。
- 赋予因子生命周期状态机属性（如 Active, Degraded, Stale, Deprecated），使历史因子能为下一轮挖掘提供启发式参考。

### 3.4 引入数据能力注册表 (Data Capability Registry)
- 构建当前系统可用数据特征的元数据层。
- 在挖掘阶段将其作为上下文注入，使 LLM 了解当前可用的数据集切面、频率、时滞限制和数据拼接规则。

### 3.5 部署持续运转调度机制 (Continuous Orchestrator)
- 将业务功能解耦为 Mining, Validation, Revalidation, Data Sync 和 Library Maintenance 等独立作业。
- 通过定时或基于事件（如新数据注入、因子过期评估）驱动系统开展无人值守的持续研究。

## 4. Consequences (影响与风险)

### Positive (收益)
- **自治运转**：系统能长效自主地挖掘、回测、清理与复用因子，大幅降低对人工手动触发的依赖。
- **有效性提升**：由于引入了多周期与多模型的相互校验，存活因子的稳健性与实盘价值将得到提高。
- **架构扩展性**：通过将调度逻辑细化与路由抽象，系统增强了针对新模型接入、新异构数据源接入的横向扩展能力。

### Negative / Risks (风险与应对)
- **算力与调用成本膨胀**：多模型并发和多周期验证会成倍增加计算及 API 开销。
  - *应对策略：严格落实漏斗型筛选机制，确保只有通过低成本初筛的高潜因子候选才能进入昂贵的回测阶段。*
- **数据对齐复杂度提升**：引入低频或存在披露时滞的指标极易引发未来函数问题。
  - *应对策略：在数据能力注册表中严格定义 `join_mode` 和 `lag_days` 约束，阻断不当数据引用。*
- **系统重构与架构膨胀风险**：一次性全面重构系统极易导致长时间不可用。
  - *应对策略：严格遵循最小可用闭环原则，将实施计划明确切割为验证层先行、模型路由次之、持续调度收尾的三个循序渐进的 Phase。*
