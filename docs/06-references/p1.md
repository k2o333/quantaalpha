# 沪深A股数据平台综合实现方案

## 一、项目概述

本方案基于MAPS范式（Model-Assemble-Process-Serve），针对单机28核CPU、32GB内存的硬件环境，构建一个高效、稳定的沪深A股数据分析平台。方案涵盖从数据下载、处理到存储的完整流程，充分利用并行计算能力，同时严格控制内存使用。实战最优版方案在原有基础上进行了深度优化，特别关注配置参数优化、存储结构简化、跨平台推理架构以及灾难恢复与监控，使其更适合长期维护的单机量化场景。

## 1. 核心理念与架构

本方案确立以下**四大核心设计原则**：

1.  **ID稳定性 (ID Stability)**：所有实体（如股票代码）的整数ID一经分配，**永久不变**。这是保证历史数据仓库长期一致性的基石。

2.  **任务分离 (Task Separation)**：严格区分**"首次全量历史构建"**和**"每日增量与修正"**两大任务。二者逻辑、目标和执行频率完全不同。

3.  **幂等性 (Idempotency)**：所有数据处理任务必须是幂等的。对同一数据范围重复执行任务，结果应与单次执行完全一致，保证系统的可恢复性。

4.  **内存外处理 (Out-of-Core Processing)**：所有操作优先采用Polars的惰性计算和流式处理，将32GB内存作为数据流转的"高速公路"，而非数据堆积的"仓库"。

### 1.1 硬件环境适配
- **CPU**: 28核处理器，支持高并发并行处理
- **内存**: 32GB RAM，需严格控制内存使用，采用流式处理
- **存储**: 大容量磁盘，采用分区存储策略

### 1.2 技术栈选择
- **数据处理**: Polars（惰性计算、内存外处理）
- **查询引擎**: DuckDB（SQL支持、分区裁剪）
- **存储格式**: Parquet（列式存储、压缩）
- **并行框架**: Python multiprocessing（利用多核）
- **元数据管理**: SQLite（状态跟踪）

## 3. 数据字典层：演进式ID管理 (核心修正)

这是对原方案重要的修正，确保ID的永久稳定性，避免ID漂移问题。引入"元数据主表"概念，实现单一数据源原则。

### 3.1 首次构建
1.  下载完整的`stock_basic`（建议包含所有上市状态`L, D, P`的股票）。
2.  生成初始的`ts_code_dict.parquet`，包含 `ts_code` 和 `ts_code_id`。此为"创世版本"。

### 3.2 每日演进 (取代原"每日重建")
此任务应在所有数据下载任务**之前**执行。

```python
import polars as pl
import pandas as pd
import logging
from config import *

def evolve_global_dictionaries():
    """
    演进式更新全局字典，确保ID的稳定性和唯一性。
    此函数应每日在主任务开始前运行。
    """
    try:
        # 1. 加载现有字典
        try:
            existing_dict_df = pl.read_parquet(DICT_DIR / 'ts_code_dict.parquet')
        except FileNotFoundError:
            # 如果是首次运行，则创建一个空的DataFrame
            existing_dict_df = pl.DataFrame({'ts_code': [], 'ts_code_id': []}, schema={'ts_code': pl.Utf8, 'ts_code_id': pl.UInt32})

        # 2. 获取最新股票列表
        latest_sh = pro.stock_basic(exchange='SSE', fields='ts_code')
        latest_sz = pro.stock_basic(exchange='SZSE', fields='ts_code')
        latest_stock_basic_df = pl.from_pandas(pd.concat([latest_sh, latest_sz]))

        # 3. 找出新增股票 (anti join)
        new_stocks_df = latest_stock_basic_df.join(
            existing_dict_df, on='ts_code', how='anti'
        )

        # 4. 如果有新股，分配新ID并追加
        if not new_stocks_df.is_empty():
            logging.info(f"发现 {len(new_stocks_df)} 只新股，正在更新字典...")
            max_id = existing_dict_df['ts_code_id'].max()
            if max_id is None:
                max_id = -1 # 处理首次运行的情况

            # 为新股生成从 max_id + 1 开始的连续ID
            new_entries_df = new_stocks_df.select('ts_code').with_row_count(
                'ts_code_id', offset=max_id + 1
            ).select(['ts_code', pl.col('ts_code_id').cast(pl.UInt32)])

            # 合并新旧字典
            updated_dict_df = pl.concat([existing_dict_df, new_entries_df])

            # 5. 覆盖写回最新、最全的字典
            # 确保目录存在
            DICT_DIR.mkdir(parents=True, exist_ok=True)
            updated_dict_df.write_parquet(DICT_DIR / 'ts_code_dict.parquet')
            logging.info("字典更新完毕。")
        else:
            logging.info("无新股，字典无需更新。")

    except Exception as e:
        logging.exception(f"更新全局字典时发生错误: {str(e)}")
        raise

# 对行业、地域等低基数维度也采用类似逻辑
```

### 4.3 速率限制系统增强 (多进程API专用限频)

原有的`RateLimiter`无法在多进程间共享。以下是增强后的实现，支持API特定限频：

```python
from multiprocessing import Manager, Lock
import time
import logging
from config import *

class AdaptiveRateLimiter:
    def __init__(self, api_name: str, max_calls: int, period_seconds: int = 60, shared_state: dict = None, lock: Lock = None):
        """
        兼容多进程的自适应速率限制器

        Args:
            api_name (str): 此限制器对应的API名称，用于状态隔离
            max_calls (int): 周期内最大调用次数
            period_seconds (int): 周期秒数
            shared_state (dict): 由Manager.dict()创建的共享字典
            lock (Lock): 由Manager.Lock()创建的共享锁
        """
        if shared_state is None or lock is None:
            raise ValueError("shared_state and lock must be provided for multiprocessing")

        self.api_name = api_name
        self.max_calls = max_calls
        self.period = period_seconds
        self.lock = lock

        # 将状态存储在共享字典中，以API名称为键
        self.shared_state = shared_state
        # 初始化该API的状态
        with self.lock:
            if self.api_name not in self.shared_state:
                self.shared_state[self.api_name] = {'timestamps': [], 'next_window': 0.0}

    def _get_api_state(self):
        # 注意：Manager.dict的修改是进程安全的，但复合操作（先读后写）不是原子的，需要锁
        return self.shared_state[self.api_name]

    def _set_api_state(self, new_state):
        self.shared_state[self.api_name] = new_state

    def wait_on_rate_limit_error(self):
        """
        当检测到速率限制错误时调用此方法。
        计算下一个窗口的开始时间并等待。
        """
        with self.lock:
            now = time.time()
            # 计算当前窗口的结束时间，即下一个窗口的开始时间
            time_in_window = now % self.period
            next_window_start = now + (self.period - time_in_window)

            # 更新共享状态
            state = self._get_api_state()
            state['next_window'] = next_window_start
            self._set_api_state(state)

            wait_time = next_window_start - now
            logging.warning(
                f"[{self.api_name}] Rate limit error detected. "
                f"Waiting for {wait_time:.2f}s until next window."
            )

        # 在锁外等待，避免阻塞其他API的锁操作
        if wait_time > 0:
            time.sleep(wait_time)

    def __call__(self):
        """
        检查是否应进行API调用，并在必要时执行等待。
        """
        wait_time = 0

        # 将锁的范围最小化，只用于读写共享状态
        with self.lock:
            now = time.time()
            state = self._get_api_state()

            # 1. 检查是否处于强制等待期
            if now < state.get('next_window', 0.0):
                wait_time = state['next_window'] - now

            # 如果不需要等待，则继续处理时间戳
            if wait_time <= 0:
                # 2. 清理过期的时间戳
                timestamps = state.get('timestamps', [])
                valid_timestamps = [t for t in timestamps if now - t < self.period]

                # 3. 检查调用次数 (此逻辑保持不变)
                if len(valid_timestamps) >= self.max_calls:
                    pass

                # 4. 记录本次调用
                valid_timestamps.append(now)
                state['timestamps'] = valid_timestamps
                self._set_api_state(state)

        # 在锁之外执行等待，这是关键！
        if wait_time > 0:
            logging.info(
                f"[{self.api_name}] In cooldown period. "
                f"Waiting for an additional {wait_time:.2f}s."
            )
            time.sleep(wait_time)

from multiprocessing import Manager, Lock

# Global variables for multiprocessing state sharing
manager = Manager()
limiter_lock = Lock()

limiter_cache = {}

def get_rate_limiter(api_name: str) -> AdaptiveRateLimiter:
    """在主进程中获取或创建限制器实例"""
    global limiter_cache
    if api_name not in limiter_cache:
        max_calls = API_LIMITS.get(api_name, API_RATE_LIMIT) # 从config获取限制
        limiter_cache[api_name] = AdaptiveRateLimiter(
            api_name=api_name,
            max_calls=max_calls,
            shared_state=limiter_shared_state,
            lock=limiter_lock
        )
    return limiter_cache[api_name]

def detect_rate_limit_error(exception: Exception) -> bool:
    """检测是否为TuShare的速率限制错误"""
    error_msg = str(exception).lower()
    rate_limit_indicators = [
        'rate limit', 'limit exceeded', 'too frequent', 'frequent', 'limited',
        'per minute', 'requests per', 'call frequency', '访问过于频繁', '请求过于频繁',
        '每分钟最多访问', '积分不足'
    ]
    return any(indicator in error_msg for indicator in rate_limit_indicators)

def safe_api_call(func, *args, api_name: str = None, **kwargs):
    """
    智能、支持多进程的API安全调用函数。
    - 自动获取对应API的速率限制器。
    - 遇到速率限制错误时，仅等待该API的窗口重置。
    - 遇到其他错误时，指数退避重试。
    """
    if api_name is None:
        # 假设有一个函数可以从func对象推断api_name
        api_name = infer_api_name(func, 'default_api')

    limiter = get_rate_limiter(api_name)

    for attempt in range(API_MAX_RETRIES):
        limiter() # 应用速率限制

        try:
            return func(*args, **kwargs)
        except Exception as e:
            if detect_rate_limit_error(e):
                # 是速率限制错误，调用限制器的等待方法
                limiter.wait_on_rate_limit_error()
                # 无需增加退避时间，直接进入下一次循环重试
                continue

            # 其他错误，进行指数退避
            if attempt == API_MAX_RETRIES - 1:
                logging.error(f"[{api_name}] API call failed after max retries: {e}")
                raise e

            wait_time = 2 ** attempt
            logging.warning(
                f"[{api_name}] API call failed with a non-rate-limit error. "
                f"Retrying in {wait_time}s... (Attempt {attempt + 1}/{API_MAX_RETRIES}). Error: {e}"
            )
            time.sleep(wait_time)

    raise Exception(f"[{api_name}] API call failed after exhausting all retries.")

def initialize_rate_limiters():
```

```python
def etl_pipeline(source_path: str, target_path: None, data_type: str, df: pl.DataFrame = None):
    """
    ETL流程：读取->转换->排序->写入（完全配置驱动）

    Args:
        source_path: 临时文件路径（仅首次构建时使用）

        # 1. 加载数据（灵活模式）
        if df is None:
            # 首次构建：从源文件读取
            lf_raw = pl.scan_parquet(source_path)
        else:
            # 增量任务：直接使用传入的df

        # 内存监控
        log_memory(f"ETL_START_{data_type}")

        # 2. 增强ID（元数据主表）
        # 检查是否有ts_code列
        if 'ts_code' in lf_raw.columns:
            lf_enhanced = lf_raw.join(
                pl.scan_parquet(DICT_DIR / 'ts_code_dict.parquet'),
                on='ts_code',
                how='left'
            ).drop('ts_code')
        else:
            # 如果没有ts_code列，直接使用原数据
            lf_enhanced = lf_raw

        # 3. 数据类型优化（统一处理日期字段）
        # 处理日线数据类型
        if data_type in ['daily', 'daily_basic', 'moneyflow', 'moneyflow_ths', 'moneyflow_dc', 'pro_bar']:
            # 日线数据：确保trade_date_int字段存在用于排序
            if 'trade_date_int' not in lf_enhanced.columns and 'trade_date' in lf_enhanced.columns:

                ])
                # 添加年分区字段
                lf_enhanced = lf_enhanced.with_columns([
                    (pl.col('trade_date_int') // 10000).cast(pl.UInt32).alias('year')
                ])
        elif data_type in ['block_trade', 'cyq_chips', 'stk_surv']:
            # 高频事件流：确保分区字段存在
            if 'trade_date_int' in lf_enhanced.columns or 'trade_date' in lf_enhanced.columns:
                # 如果trade_date是字符串格式，先转换为整数
                if 'trade_date' in lf_enhanced.columns:

                        pl.col('trade_date').cast(pl.Utf8).str.replace('-', '').cast(pl.UInt32).alias('trade_date_int')
                    ])
                # 添加分区字段

                    (pl.col('trade_date_int') // 10000).cast(pl.UInt32).alias('year')
                ])
            elif 'ann_date_int' in lf_enhanced.columns or 'ann_date' in lf_enhanced.columns:
                # 处理公告日期字段
                if 'ann_date' in lf_enhanced.columns:
                    lf_enhanced = lf_enhanced.with_columns([
                        pl.col('ann_date').cast(pl.Utf8).str.replace('-', '').cast(pl.UInt32).alias('ann_date_int')
                    ])
                # 添加年月分区字段
                lf_enhanced = lf_enhanced.with_columns([
                    (pl.col('ann_date_int') // 100).cast(pl.UInt32).alias('year_month')
                ])

        # 内存监控
        log_memory(f"POST_TRANSFORM_{data_type}")

        # 4. 排序（根据数据类型）
        sort_cols = {
            'daily': ['ts_code_id', 'trade_date_int'],  # 包含日线数据
            'daily_basic': ['ts_code_id', 'trade_date_int'],

            'disclosure_date': ['ts_code_id', 'ann_date_int'],

        # 内存监控
        log_memory(f"POST_SORT_{data_type}")

        # 5. 写入（根据配置）
        base_path, partition_granularity = PARTITION_CONFIG[data_type]

        if partition_granularity == PartitionGranularity.YEAR:
            atomic_partitioned_sink(lf_sorted, base_path, partition_by=['year'])
        elif partition_granularity == PartitionGranularity.YEAR_MONTH:

            atomic_unpartitioned_sink(lf_sorted, output_path)

        logging.info(f"ETL完成: {data_type} -> {base_path}")
        log_memory(f"ETL_END_{data_type}")

def atomic_unpartitioned_sink(lazy_frame, target_path):
```

```python
def normalize_dtypes_for_concat(dataframes):
    """
    Normalize data types across multiple dataframes to allow concatenation
    This handles cases where the same column has different types across periods
    """
    if not dataframes:
        return dataframes

    if len(dataframes) == 1:
        return dataframes

    # 获取参考模式（第一个DataFrame）
    ref_cols = set(dataframes[0].columns)
    normalized_dfs = []

    for df in dataframes:
        # 确保所有DataFrame有相同的列
        for col in ref_cols:
            if col not in df.columns:
                # 如果参考DF有但当前DF没有，则添加空列
                df = df.with_columns(pl.lit(None).alias(col))

        # 转换数据类型以与参考DF保持一致
        for col in ref_cols:
            ref_dtype = dataframes[0].schema.get(col)
            actual_dtype = df.schema.get(col)

            if ref_dtype and actual_dtype and ref_dtype != actual_dtype:
                try:
                    # 尝试将当前DF的列转换为参考类型
                    df = df.with_columns(pl.col(col).cast(ref_dtype, strict=False))
                except:
                    # 如果转换失败，将参考DF和当前DF的列都转换为String类型
                    df = df.with_columns(pl.col(col).cast(pl.Utf8))
                    dataframes[0] = dataframes[0].with_columns(pl.col(col).cast(pl.Utf8))

        normalized_dfs.append(df)

    return normalized_dfs
```

## 7. 任务执行工作流 (核心修正)

明确分离两大工作流，逻辑更清晰。

### 7.1 工作流 A：首次全量历史数据构建 (执行一次)
**目标**：下载并处理从市场开始到昨天的所有历史数据。

1.  **准备**：执行`evolve_global_dictionaries()`创建初始字典。初始化SQLite元数据数据库。
2.  **财务数据**：
    *   **按报告期遍历**，从`19900331`至今。
    *   使用`income_vip`, `balancesheet_vip`等接口，每次获取一个报告期的**全部公司**数据。
    *   下载 -> 暂存 -> ETL (使用字典) -> 写入最终**单个文件**（不再分区）。
    *   **实现断点续传**：每处理完一个报告期，就在SQLite中记录状态。
3.  **行情数据**：
    *   **按股票代码遍历**。
    *   对每个股票，获取其从上市日期到昨天的所有`daily`, `daily_basic`, `stk_factor`等数据。
    *   同样遵循 下载 -> 暂存 -> ETL -> 写入最终分区的流程。

### 7.2 工作流 B：每日增量与修正任务 (每日执行)
**目标**：获取昨日的新数据，并修正近期可能变更的数据。

1.  **元数据更新**：首先运行`evolve_global_dictionaries()`和`update_stock_basic_with_ids()`，确保字典是最新状态。
2.  **"回补"逻辑 (Catch-up)**：
    *   从SQLite读取各表（如`daily`）的`last_update_date`。
    *   计算需要下载的日期范围：`start_date = last_update_date + 1天`, `end_date = 昨天`。
3.  **行情数据更新**：
    *   遍历日期范围，每日下载`daily`, `daily_basic`, `moneyflow`, `stk_factor`等数据。
    *   ETL处理后，**覆盖写入**到对应的日期分区中，实现幂等性。
    *   **在循环内部，每成功处理一天，就更新一次SQLite中的`last_update_date`**，确保任务失败可从断点恢复。
4.  **财务数据修正 (简化策略)**：
    *   无需每日检查。可设置为**每周末或每月初**执行。
    *   **策略**：重新下载并覆盖**最近2-3个季度**的所有财务报表数据（如`20231231`, `20240331`, `20240630`）。这能以极小的开销捕获绝大多数财报修订，且实现简单。

### 9.3 分区策略
- **行情数据**: 按年分区（`year=2023`）
- **日线数据**: 按年分区（`year=2023`），使用后复权价格
- **财务数据**: **不分区**，单个文件存储（简化维护）
- **资金数据**: 按年分区（`year=2023`）
- **高频事件流数据**（如`block_trade`）：按月分区（`year_month=202301`），避免单文件过大导致性能问题
- **筹码分布数据**（`cyq_chips`）：按年分区（`year=2023`），避免单文件OOM
- **低频事件流数据**（如`stk_surv`）：单文件存储，定期compact避免碎片化
- **事件数据**（如`forecast`, `express`）：单文件覆盖写入（公告后覆盖）
- **公司信息**: 不分区（数据量小）

### 9.4 排序策略
- **日线数据**: `ts_code_id, trade_date_int`
- **财务数据**: `ts_code_id, end_date`
- **资金数据**: `ts_code_id, trade_date_int`
- **保证同一实体的数据在物理上连续存储**

### 9.5 压缩策略
- 使用ZSTD压缩算法，在压缩率和解压速度间取得平衡

```python
def atomic_partitioned_sink(lazy_frame, target_root, partition_by):
    """对分区数据进行原子化写入"""
    temp_root = f"{target_root}_tmp_{os.getpid()}_{time.time()}"

    try:
        # 1. 写入到临时根目录
        lazy_frame.sink_parquet(temp_root, partition_by=partition_by, compression=COMPRESSION_TYPE)

        # 2. 确保目标目录存在
        os.makedirs(target_root, exist_ok=True)

        # 3. 将生成的分区目录逐个移动到最终位置
        for item in os.listdir(temp_root):
            src_path = os.path.join(temp_root, item)
            dst_path = os.path.join(target_root, item)

        logging.info(f"原子化写入完成: {target_root}")

    except Exception as e:
        logging.exception(f"原子化写入失败: {str(e)}")
        raise

    finally:
        # 4. 清理临时根目录
        if os.path.exists(temp_root):
            shutil.rmtree(temp_root)
```
