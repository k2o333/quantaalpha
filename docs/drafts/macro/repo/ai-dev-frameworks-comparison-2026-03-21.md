# AI 开发规范框架比较报告

**报告日期**: 2026-03-21  
**分析对象**: 5 个主流 AI 开发规范系统  
**分析来源**: DeepWiki 官方文档

---

## 执行摘要

本报告对五个主流 AI 开发规范系统进行了全面比较分析：`everything-claude-code`、`get-shit-done (GSD v1)`、`gsd-2`、`superpowers` 和 `OpenSpec`。这些系统代表了当前 AI 辅助软件开发领域的最佳实践和不同设计哲学。

**核心发现**:
- **功能全面性**: `everything-claude-code` 提供最全面的功能集（25+ 代理、108+ 技能、57+ 命令）
- **阶段控制**: `get-shit-done` 提供最严格的阶段门控工作流
- **自主化程度**: `gsd-2` 在自主开发和成本优化方面领先
- **技能系统**: `superpowers` 的可组合技能系统最具可扩展性
- **灵活性**: `OpenSpec` 的增量规范系统最适合迭代开发和现有项目

## 第一章：项目概述

### 1.1 everything-claude-code

| 属性 | 描述 |
|:-----|:-----|
| **仓库** | `affaan-m/everything-claude-code` |
| **定位** | Claude Code 综合增强系统 |
| **核心目标** | 优化 token 使用、强制执行代码质量、跨会话持久化内存、实现自主工作流 |
| **文档规模** | 16 个主章节，80+ 子主题 |

**核心理念**:
1. **代理优先 (Agent-First)**: 将领域任务委托给专业代理
2. **测试驱动 (Test-Driven)**: 在实现之前编写测试，要求 80% 以上的覆盖率
3. **安全优先 (Security-First)**: 绝不妥协于安全性，验证所有输入
4. **不可变性 (Immutability)**: 始终创建新对象，从不修改现有对象
5. **先计划后执行 (Plan Before Execute)**: 在编写代码之前规划复杂功能

---

### 1.2 get-shit-done (GSD v1)

| 属性 | 描述 |
|:-----|:-----|
| **仓库** | `gsd-build/get-shit-done` |
| **定位** | 元提示、上下文工程和规范驱动的开发系统 |
| **核心目标** | 将 AI 编码助手转化为可靠、结构化的开发工具 |
| **文档规模** | 11 个主章节，50+ 子主题 |

**核心理念**:
1. **多运行时支持**: 支持 6 种 AI 运行时，"一次编写，随处部署"
2. **阶段式工作流**: 强制执行严格的"讨论 → 计划 → 执行 → 验证"生命周期
3. **薄编排器模式**: 轻量级协调器生成具有全新上下文的专用代理
4. **文件式状态管理**: 所有状态持久化在 `.planning/` 目录
5. **原子 Git 提交**: 每个任务完成后立即生成独立提交

---

### 1.3 gsd-2

| 属性 | 描述 |
|:-----|:-----|
| **仓库** | `gsd-build/gsd-2` |
| **定位** | GSD 的下一代版本，自主开发引擎 |
| **核心目标** | 实现高效、可扩展和可控的自主开发 |
| **文档规模** | 10 个主章节，40+ 子主题 |

**核心理念**:
1. **从结果而非功能开始**: 在开始任何工作之前定义"完成"的标准
2. **垂直切片而非水平分层**: 构建端到端的垂直切片
3. **一天规则**: 如果一个任务需要超过一天的时间，则应进一步分解
4. **风险优先探索**: 首先识别最困难或最不确定的部分
5. **接口优先设计**: 在构建组件之前定义组件之间的契约
6. **MECE 分解**: 任务应相互独立且共同穷尽

---

### 1.4 superpowers

| 属性 | 描述 |
|:-----|:-----|
| **仓库** | `obra/superpowers` |
| **定位** | 为编码代理设计的软件开发工作流系统 |
| **核心目标** | 基于可组合的"技能"确保代理有效利用最佳实践 |
| **文档规模** | 11 个主章节，45+ 子主题 |

**核心理念**:
1. **测试驱动开发 (TDD)**: 无论是代码还是技能的创建，都强调先写测试
2. **系统性而非随意性**: 通过流程而非猜测来解决问题
3. **降低复杂性**: 以简洁为主要目标
4. **证据而非断言**: 在宣布成功之前进行验证
5. **YAGNI**: 避免过度工程
6. **DRY**: 减少重复
7. **1% 规则**: AI 代理必须调用 `Skill` 工具，即使只有"1% 的可能性"该技能适用

---

### 1.5 OpenSpec

| 属性 | 描述 |
|:-----|:-----|
| **仓库** | `Fission-AI/OpenSpec` |
| **定位** | AI 原生、规范驱动的软件开发系统 |
| **核心目标** | 通过结构化规范促进 AI 编码助手和开发人员之间的协作 |
| **文档规模** | 9 个主章节，40+ 子主题 |

**核心理念**:
1. **灵活而非僵化**: 没有阶段门，可以随时进行工作
2. **迭代而非瀑布式**: 在构建过程中学习，在改进过程中完善
3. **简单而非复杂**: 轻量级设置，最小化仪式
4. **优先考虑现有项目**: 适用于现有代码库，而不仅仅是全新项目

## 第三章：工作流详细比较

### 3.1 everything-claude-code 工作流

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   研究阶段   │ →  │   构思阶段   │ →  │   计划阶段   │
│  Research   │    │  Brainstorm │    │    Plan     │
└─────────────┘    └─────────────┘    └─────────────┘
       ↑                                       │
       │                                       ↓
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   审查阶段   │ ←  │   优化阶段   │ ←  │   执行阶段   │
│   Review    │    │  Optimize   │    │  Execute    │
└─────────────┘    └─────────────┘    └─────────────┘
```

**阶段详情**:

| 阶段 | 主要活动 | 使用代理/技能 | 输出物 |
|:-----|:---------|:--------------|:-------|
| 研究 | GitHub 搜索、文档查阅、Exa 网络研究、包注册表检查 | `researcher`, `search` | 可重用库列表 |
| 构思 | 功能设计、替代方案探索 | `brainstormer` | 设计方案 |
| 计划 | 实现计划、依赖识别、风险分解 | `planner`, `architect` | 规划文档、阶段分解 |
| 执行 | TDD (RED-GREEN-IMPROVE)、80%+ 覆盖率 | `tdd-guide`, `executor` | 代码 + 测试 |
| 优化 | 代码重构、性能优化 | `optimizer` | 优化后的代码 |
| 审查 | 质量审查、安全检查 | `code-reviewer` | 审查报告、问题列表 |

**关键实践**:
- 强制 80%+ 测试覆盖率
- 安全审查清单（无硬编码秘密、输入验证、SQL 注入防护等）
- 代码质量标准（函数<50 行、文件<800 行、嵌套<4 级）

**硬门 (Hard-Gate) 机制**:

| 硬门位置 | 阻止条件 | 需要条件 |
|:---------|:---------|:---------|
| 头脑风暴后 | 禁止实施 | 用户批准设计文档 |
| 编写计划后 | 禁止执行 | 计划文档审查通过 |
| 任务之间 | 禁止继续 | 代码审查关键问题解决 |

### 8.4 superpowers

**优势**:
- ✅ 技能系统可组合、可扩展
- ✅ 硬门质量保证机制
- ✅ 子代理隔离防污染
- ✅ 两阶段审查（规范 + 代码）
- ✅ 多平台支持好

**局限**:
- ❌ 技能创建需要投入
- ❌ 硬门可能降低速度
- ❌ 无持续学习机制

### 8.5 OpenSpec

**优势**:
- ✅ 灵活性最高
- ✅ 增量规范系统
- ✅ 最适合现有项目
- ✅ AI 驱动文档
- ✅ 学习曲线最低

**局限**:
- ❌ 灵活性可能导致纪律松散
- ❌ 规范维护成本
- ❌ 无自主执行能力
