## 二、发现的错误

### 错误 1: M001-CONTEXT.md Bug 2 行号描述不准确

**位置**: M001-CONTEXT.md:17

**文档描述**:
```
Bug 2: LLM 空响应导致 JSON 解析崩溃
- 位置: quantaalpha/llm/client.py:1047-1051
- 问题: 当 resp 为空字符串时，JSON 提取逻辑产生无效切片
```

**实际代码** (`client.py:1047-1051`):
```python
if json_mode or reasoning_flag:
    # Extract JSON part
    json_start = resp.find('{')
    json_end = resp.rfind('}') + 1
    resp = resp[json_start:json_end]
```

**问题**: 文档将这段代码描述为"空响应检查逻辑"，实际上这是 **JSON 提取逻辑**。当 `resp` 为空时，`resp.find('{')` 返回 `-1`，`resp.rfind('}')` 返回 `-1`，导致 `resp = resp[-1:0]`（空字符串），后续 `json.loads("")` 抛出 `JSONDecodeError`。根因描述正确，但代码定位和功能描述不精确。

此 Bug 与 Bug 1-4 无关，是独立的类型处理错误。文档应将其列为第 5 个 Bug 或在"其他问题"章节提及。

### 错误 4: S01-PLAN.md 验证步骤 grep 模式无效

**位置**: S01-PLAN.md:36, 43, 50

**文档描述** (以 T01 为例):
```
Verify: grep -n "logger.warning.*%s" third_party/quantaalpha/quantaalpha/llm/client.py | head -5 返回空
```

**实际问题**: `logger.warning()` 在 `client.py:69-74` 是**多行调用**：
```python
logger.warning(
    "Tokenizer lookup failed for model %s; falling back to %s. reason=%s",
    model,
    DEFAULT_FALLBACK_TOKENIZER,
    reason,
)
```

标准 `grep` 不支持跨行匹配，`"logger.warning.*%s"` 模式无法匹配此多行调用。因此即使 Bug 1 未修复，验证命令也会错误地"返回空"（通过验证）。

**正确验证方式**: 应使用 `grep -Pzo` 或 `rg --multiline`，或改为检查修复后的 f-string 模式：
```bash
grep -n 'logger.warning(f"' third_party/quantaalpha/quantaalpha/llm/client.py
```

### 错误 5: S03-PLAN.md 验证步骤 grep 模式无效

**位置**: S03-PLAN.md:35

**文档描述**:
```
Verify: grep -n "\\\\n" third_party/quantaalpha/quantaalpha/llm/client.py | grep -A2 -B2 "fixed_resp" 找到转义逻辑
```

**问题**: `\\\\n` 在 shell grep 中会被解释为字面量 `\\n`（两个字符：反斜杠+n），而非实际换行符。实际代码中需要匹配的是对 `fixed_resp` 变量中**实际换行字符**的替换操作，但 `\\\\n` 模式无法匹配实际换行符。此验证命令即使修复后也不会按预期工作。

### 错误 7: S01 与 S02 修复方案存在逻辑冲突

**位置**: S01-PLAN.md:56, S02-PLAN.md:34

**S01 方案**: 在 `_create_chat_completion_inner_function` 中添加空响应检查：
```python
if not resp or not resp.strip():
    raise RuntimeError(f"LLM returned empty response for model {model}")
```

**S02 方案**: 将 `while True` 改为有上限的重试循环。

**冲突**: S01 的 `raise RuntimeError` 会直接终止程序，不会给 S02 的重试逻辑任何机会。正确的修复顺序应该是：
1. S01 只修复 logger 参数问题
2. S02 添加重试上限
3. 空响应检查应在 S02 的重试循环内部进行，失败时 `continue` 而非 `raise`

文档中 S01 的"空响应检查逻辑"应改为在 S02 修复后的循环内实现，或改为 `logger.warning` + `continue` 而非直接抛异常。
