# M001 文档事实核查补充报告

## 结论

`.gsd/milestones/M001` 下面的文档不是完全正确。基于两份终端输出和 `third_party/quantaalpha/run.sh` 实际调用链，`M001` 至少存在 4 类文档问题：

1. 把问题总数写成“4 个关联 Bug”是不准确的，`20260321_214610.txt` 已经显示出第 5 个实际故障。
2. Bug 1 的定位范围写得过大，把没有在当前调用链中被证实的点也写成了“核心问题”。
3. Bug 3 的故障阶段写错了，死循环发生在 `factor_construct`，不是文档中笼统写的 `proposal` 阶段。
4. `S03` 被写成“最后一个修复”不成立，因为第二份日志里还有未被 M001 纳入的真实故障。

## 事实依据

`run.sh` 实际执行的是：

- `third_party/quantaalpha/run.sh:121-124`
- `third_party/quantaalpha/quantaalpha/pipeline/factor_mining.py:378-386`
- `third_party/quantaalpha/quantaalpha/pipeline/loop.py:172-187`

也就是：

1. `run.sh` 调用 `quantaalpha mine`
2. `factor_mining.run_evolution_loop()` 先调用 `generate_parallel_directions()`
3. 然后进入 `AlphaAgentLoop.factor_propose()`
4. 再进入 `AlphaAgentLoop.factor_construct()`

两份日志里与 M001 直接相关的已观测事实：

- `20260321_214610.txt` 21:46:13 先出现 `Planning LLM call failed (attempt 1): RDAgentLog.warning() takes 2 positional arguments but 5 were given`
- `20260321_190705.txt` 在 `factor_propose took 0.45s` 之后，开始持续刷 `quantaalpha.factors.proposal:_convert_with_history_limit:491 - JSON parse failed ... retrying...`
- `20260321_214610.txt` 21:47:50 出现 `Consistency check error: 'dict' object has no attribute 'replace'`
- `20260321_214610.txt` 21:48:48 起多次出现 `JSON fix failed: Invalid control character`

## 文档错误明细

### 1. “4 个关联 Bug”结论不完整

文档位置：

- `.gsd/milestones/M001/M001-CONTEXT.md:9-29`
- `.gsd/milestones/M001/M001-ROADMAP.md:3-10`

问题：

文档把 M001 收敛为 4 个 Bug，但第二份日志已经出现了文档完全未覆盖的故障：

- `20260321_214610.txt:208` 左右：`Consistency check error: 'dict' object has no attribute 'replace'`

这不是噪声，而是当前 `run.sh` 调用链上的真实错误。源码链路也能对上：

- `third_party/quantaalpha/quantaalpha/factors/proposal.py:517-544`
- `third_party/quantaalpha/quantaalpha/factors/regulator/consistency_checker.py:168-175`
- `third_party/quantaalpha/quantaalpha/factors/regulator/consistency_checker.py:259-265`
- `third_party/quantaalpha/quantaalpha/factors/regulator/consistency_checker.py:472-488`

`check_and_correct()` 接受并返回的 `corrected_expression` 被假定为字符串，但日志里 LLM 返回了一个字典结构，后续复杂度检查里执行 `expression.replace(" ", "")` 时直接炸掉。  
因此，“M001 覆盖了全部关键问题”这个判断不成立。

### 2. Bug 1 的定位范围写大了

文档位置：

- `.gsd/milestones/M001/M001-CONTEXT.md:11-14`
- `.gsd/milestones/M001/slices/S01/S01-PLAN.md:32-57`

问题：

文档把 Bug 1 的位置写成：

- `quantaalpha/llm/client.py:69-74`
- `client.py:667`
- `backtest/universe.py:111`

但结合当前调用链和日志，真正被证实触发的是 `client.py:69-74`，不是另外两个点。

已证实的触发链：

- `factor_mining.py:378-386` 调用 `generate_parallel_directions()`
- `planning.py:133-142` 调用 `APIBackend().build_messages_and_create_chat_completion(...)`
- `llm/client.py:543-553` 初始化 tokenizer fallback 时会走 `log_tokenizer_fallback_once()`
- `llm/client.py:69-74` 使用了多参数 `logger.warning(...)`

对应日志：

- `20260321_214610.txt:25` 左右：`Planning LLM call failed (attempt 1): RDAgentLog.warning() takes 2 positional arguments but 5 were given`

相反：

- `client.py:667` 只有在传入未知 `task_type` 时才会触发，本次 planning 调用没有传这个参数
- `backtest/universe.py:111` 属于股票池日期解析，当前两份日志没有出现 `as_of_date` 解析失败，也没有证据表明它参与了本次首轮故障

所以，S01 把 `universe.py` 和 `client.py:667` 一并定义为本问题的主修复点，证据不足。

### 3. Bug 3 的故障阶段写错了

文档位置：

- `.gsd/milestones/M001/M001-CONTEXT.md:9`
- `.gsd/milestones/M001/M001-CONTEXT.md:21-24`
- `.gsd/milestones/M001/M001-ROADMAP.md:9`
- `.gsd/milestones/M001/slices/S02/S02-PLAN.md:3-4`

问题：

文档多处把问题表述成“proposal 阶段死循环”，但从当前代码和日志看，真正发生无限重试的位置在 `factor_construct()` 内部的 `AlphaAgentHypothesis2FactorExpression._convert_with_history_limit()`，不是 `factor_propose()`。

源码：

- `third_party/quantaalpha/quantaalpha/pipeline/loop.py:172-178` 是 `factor_propose`
- `third_party/quantaalpha/quantaalpha/pipeline/loop.py:182-187` 是 `factor_construct`
- `third_party/quantaalpha/quantaalpha/factors/proposal.py:483-492` 的 `while True` 属于 factor expression conversion

日志也吻合这个结论：

- `20260321_190705.txt` 先记录 `factor_propose took 0.45s`
- 随后才开始持续刷 `quantaalpha.factors.proposal:_convert_with_history_limit:491 - JSON parse failed ... retrying...`

因此，M001 对故障阶段的描述不精确。更准确的说法应当是：

- `factor_propose` 会因为空响应降级到 fallback hypothesis
- 真正导致刷屏重试/卡住的是后续 `factor_construct` 中的无限 `while True`

### 4. Bug 3 的触发条件写窄了

文档位置：

- `.gsd/milestones/M001/M001-CONTEXT.md:23`
- `.gsd/milestones/M001/slices/S02/S02-PLAN.md:32-35`

问题：

文档把 Bug 3 写成“当 LLM 持续返回空响应时陷入死循环”，但代码实际条件更宽：

- `third_party/quantaalpha/quantaalpha/factors/proposal.py:488-492`

这里只要 `robust_json_parse(resp)` 抛 `json.JSONDecodeError` 就会 `continue`，不限于空响应。

第二份日志已经证明，控制字符导致的无效 JSON 同样会不断触发这条重试路径：

- `20260321_214610.txt:283-284` 左右：`Invalid control character`

所以，文档对 Bug 3 的描述过窄。它不是“只对空响应敏感”，而是“对任意持续 JSON 解析失败都没有退出条件”。

### 5. “S03 是最后一个修复”不成立

文档位置：

- `.gsd/milestones/M001/slices/S03/S03-PLAN.md:23-27`
- `.gsd/milestones/M001/M001-ROADMAP.md:31-36`

问题：

S03 写成“无，这是最后一个修复”，但 `20260321_214610.txt` 里至少还有一个独立于控制字符的未覆盖故障：

- `Consistency check error: 'dict' object has no attribute 'replace'`

这说明即使按 M001 计划修完 S01-S03，系统也不一定达到文档声称的 Definition of Done。  
换句话说，M001 的完成条件写早了。

## 建议如何修正文档

如果后续要修正文档而不是修代码，建议至少做 4 个调整：

1. 把“4 个关联 Bug”改成“已确认 4 个高优先级 Bug，另有至少 1 个 consistency correction 数据形状问题未纳入本里程碑”。
2. 把 Bug 1 的核心位置收缩到当前调用链已证实的点，`client.py:667` 和 `backtest/universe.py:111` 最多列为“同类隐患”，不要列成当前事故主因。
3. 把 Bug 3 的故障阶段从笼统的 `proposal` 改成 `factor_construct / hypothesis2experiment conversion`。
4. 把 S03 的 “What remains: 无” 和 DoD 改成条件式表述，明确说明还需覆盖 consistency correction 的类型校验问题。

## 最终判断

M001 文档不是完全错误，但存在明显的“范围收缩过头”和“调用链归因不够精确”问题。  
如果直接按当前 M001 执行，能覆盖一部分真实问题，但会漏掉 `20260321_214610.txt` 已经暴露出来的 consistency correction 数据类型错误，且会把一部分未在当前事故中证实的点误写成主问题。
