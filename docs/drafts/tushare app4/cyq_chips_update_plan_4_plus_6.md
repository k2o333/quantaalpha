# cyq_chips 最合适的 4+6 组合方案

## 背景

`cyq_chips` 的问题不是单纯的 Tushare 偶发报错，而是下面两件事叠加：

1. 单个股票任务太大，常常需要 offset 翻很多页。
2. offset 中途报错时，当前实现会把已经拉到的前几页先落盘。

这样会产生一种危险状态：

- 某个 `ts_code + 日期范围` 任务只写入了前半段数据；
- 但存储里已经出现了这个股票这段日期的数据；
- 之后重跑时，容易被误当成"这段已经下载过"；
- 结果这次中断点之后的页很难再补回来。

对 `cyq_chips` 来说，最合适的短中期方案是：

- 方案 4：offset 任务失败时，不保存当前任务的残留页。
- 方案 6：把单个任务切小，降低一次失败造成的损失。

这套方案的目标不是做精确断点续传，而是：

- 不再制造新的半截脏数据；
- 让失败只影响一个小窗口；
- 让 `--update` 成为可接受、可重复执行的稳定路径。

## 为什么选 4+6

### 方案 4 的价值

当前 offset 执行逻辑里，异常分支会先保存 `all_data`，再抛错。

这一步正是半截数据的来源。

对于 `cyq_chips` 这类单任务数据量大的接口，宁可让当前任务整段失败重跑，也不要把半段数据写入正式数据区。

因此，`cyq_chips` 最需要的第一件事是：

- offset 失败时，当前任务已经拉到但尚未完整完成的页，不要保存。

### 方案 6 的价值

即使做了方案 4，如果一个任务还是"单股全历史"，那么：

- 一次报错就浪费很多已拉取页；
- 重试成本很高；
- `--update` 虽然可用，但效率和稳定性都不好。

所以必须进一步切小任务：

- 把每个股票的下载范围拆成更小的时间窗口；
- 必要时降低单页 limit；
- 用更多小任务换更低失败成本。

对 `cyq_chips` 来说，这是比"单股全历史 + offset"更稳的运行方式。

## 当前代码中的关键位置

### offset 异常时保存残留页

文件：

- `app4/core/pagination_executor.py`

问题点：

- `_execute_single_request()` 的 offset 分页异常处理中，当前会先执行 `save_callback(interface_name, all_data)`，然后抛出异常。

结果：

- 当前任务虽然失败，但部分页面已经进入正式数据。

### cyq_chips 当前任务过大

文件：

- `app4/config/interfaces/cyq_chips.yaml`

当前配置特点：

- `pagination.mode: stock_loop`
- `time_range.window: 9999d`
- `offset.limit: 5999`

这等价于：

- 对每只股票尽量一次拉很长时间；
- 如果数据量超大，就依赖 offset 翻很多页。

对一个经常偶发 error 的上游接口，这个任务粒度太大。

## 推荐落地方案

### 一、先改方案 4

#### 原则

对于 `cyq_chips` 这类需要 offset 的接口：

- offset 循环只要没有正常跑到最后一页，就视为当前任务失败；
- 当前任务已拉到但还没确认完整的数据，不进入正式数据区；
- 最小保存单位不是单页，而是"一个股票的一个日期窗口"。

也就是说：

- 对 `cyq_chips`，最小提交单元应当是 `ts_code + start_date + end_date`；
- 只要这个窗口没有完整跑完，就整窗不保存；
- 不是"先保存，失败后再移出"，而是"未完整完成就根本不保存"。

#### 推荐做法

修改 `app4/core/pagination_executor.py`：

- 在 `_execute_single_request()` 的 offset 异常分支中：
  - 去掉异常时对 `all_data` 的保存；
  - 直接抛出异常；
  - 让上层把这次 `ts_code + 时间窗口` 当成失败任务；
  - 只有 offset 正常跑到结束条件时，才允许保存这个窗口的数据。

#### 为什么这样更合适

因为 `cyq_chips` 的问题不是"失败后什么都没留下"，而是"失败后留下了半截数据"。

只要半截窗口不进入正式区，后续重跑就不会被这次失败污染。

相反，如果采用"先保存，失败后再移出不完整数据"的思路，会带来额外复杂度：

- 需要准确识别这次失败到底写入了哪些页；
- 需要保证删除时不会误删历史正常数据；
- 异步写入、去重、批量保存都会让回滚逻辑变得更脆弱。

因此，对当前版本来说，最稳妥的策略不是"回滚半截数据"，而是：

- 半截窗口根本不进入正式存储。

### 二、再改方案 6

#### 原则

把 `cyq_chips` 的单任务缩小到"失败后可以接受重跑"的大小。

#### 推荐配置

建议先从下面这个保守版本开始：

```yaml
pagination:
  enabled: true
  mode: stock_loop
  skip_time_range_in_full_history: false
  time_range:
    enabled: true
    window: 90d
    stop_on_empty: 90
  offset:
    enabled: true
    limit: 3000
```

如果 90 天仍然不稳，再收紧到：

- `window: 60d`
- `limit: 2000`

如果 90 天很稳，再尝试：

- `window: 120d`
- `limit: 3000`

#### 推荐调参顺序

优先调整顺序：

1. 先缩短 `window`
2. 再降低 `limit`

原因：

- `window` 决定单个任务的业务范围；
- `limit` 决定单页大小和单次响应压力；
- 先切时间，再控分页，通常更容易找到稳定点。

## 推荐运行方式

### 不再推荐

不再推荐对 `cyq_chips` 使用：

- 非 `--update`
- 单股全历史
- 依赖超长时间范围 + 多页 offset 一次跑完

这是最容易在中途报错后留下半截数据的路径。

### 推荐

推荐改成：

1. 使用 `--update`
2. 使用更小的 `time_range.window`
3. 接受少量重复下载，由主键去重兜底

核心思想是：

- 允许重复，不允许半截成功。

这比追求"一次把整只股票从头到尾完美拉完"更适合不稳定的 Tushare 接口。

## 对 cyq_chips 的具体建议

### 配置建议

建议把 `cyq_chips.yaml` 调整到类似下面的方向：

- `time_range.window: 60d` 或 `90d`
- `offset.limit: 2000` 或 `3000`

不建议继续使用：

- `window: 9999d`
- `limit: 5999`

### 运行建议

推荐把 `cyq_chips` 的更新理解为"滚动修复 + 增量前进"：

- 每次跑 `--update`
- 允许回看最近一段时间
- 用重复下载覆盖偶发失败留下的风险

如果需要一次性修复历史问题，建议单独做一次专项回刷，而不是指望日常 `--update` 自动修好旧的半截任务。

## 这套方案解决什么，不解决什么

### 能解决

- 不再生成新的半截 offset 脏数据
- 单次失败影响范围明显变小
- `cyq_chips` 使用 `--update` 的稳定性明显提高
- 重跑时更容易补回失败窗口

### 不能彻底解决

- 已经写入存储的历史半截数据，不会自动消失
- 它不是严格意义上的断点续传
- 如果上游接口长期极不稳定，任务仍然可能失败，只是失败成本更低

## 为什么这是当前最合适的方案

相比增加一整套 offset checkpoint 或 staging 机制，`4+6` 有几个现实优势：

1. 改动小
2. 风险可控
3. 能快速止血
4. 对 `cyq_chips` 这类大任务接口收益立刻可见

它不是最终形态，但非常适合作为当前版本的主方案。

如果后续还要继续提高稳定性，再考虑：

- 任务级 checkpoint
- offset 完整性元数据
- staging + promote 的两阶段提交

## 最终结论

对 `cyq_chips`，当前最合适的做法是：

1. 先落实方案 4
2. 再落实方案 6
3. 然后把日常运行路径切换到 `--update`

用一句话概括：

- 不要再让单股全历史的大 offset 任务失败后留下半截数据；
- 要把任务切小，让失败可重跑；
- 宁可重复下载，也不要部分成功。
