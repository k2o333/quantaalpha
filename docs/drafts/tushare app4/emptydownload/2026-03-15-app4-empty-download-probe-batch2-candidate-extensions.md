# App4 空结果探测记录方案 第二批

## 范围

这一批不是立即实施，而是“可以继续评估”的候选接口。

接口：

- `stk_factor_pro`
- `moneyflow_dc`

## 为什么放第二批

这两个接口虽然当前是：

- `reverse_date_range`

但业务语义更接近：

- `ts_code + trade_date` 是否存在

也就是说，它们有潜力收敛到与第一批类似的股票级检测键，但当前代码还没有这样建模。

## 当前不直接实施的原因

### 1. 现有检测逻辑不是股票级键存在

当前 `reverse_date_range` 大多还是按：

- 日期范围覆盖率

来决定是否跳过，而不是按：

- `ts_code + trade_date`

如果不先改检测逻辑，只是把空 probe 写进去，收益有限。

### 2. 同表占位的读污染风险更高

这类接口通常已有较强的下游分析使用场景。

一旦同表写 probe，就必须确保所有读取都默认过滤掉 `is_empty_probe = 1`。

## 候选设计

### 检测键

对这两个接口，候选检测键都建议是：

- `ts_code + trade_date`

### 前置改造

必须先做：

1. 把覆盖率检测从 `date_range` 逻辑切到股票级存在检测
2. 明确请求语义确实稳定映射到 `ts_code + trade_date`
3. 验证删除 parquet 后重新补数不会被旧 probe 阻塞

## 不建议抢跑的实现

暂不建议：

- 直接在 `reverse_date_range` 路径里统一加空占位
- 用 `trade_date` 单键表示所有空结果
- 用“范围下载过”替代“单键探测过”

## 进入实施的条件

只有当下面条件成立时，第二批才进入编码：

1. 第一批已在 `cyq_chips` 跑通
2. `CoverageManager` 已有稳定的更细股票级检测策略
3. 默认读取过滤机制已经完成
4. 真实记录覆盖 probe 的优先级已验证

## 当前结论

第二批保留为候选扩展，不应与第一批并行开发。

当前最合理的做法是：

- 先把它们列入设计
- 不立即在代码里启用空占位
