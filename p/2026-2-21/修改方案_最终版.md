# App4 日期锚定接口参数传递修复方案（最终版）

## 修改目标

**仅影响**：`reverse_date_range` 模式 + `is_date_anchor: true` 的接口（目前只有 `cyq_perf`）

**不影响**：
- `stock_loop` 模式的日期锚定接口（top10_holders, stk_rewards, disclosure_date, dividend, pledge_stat）
- 所有其他普通接口

---

## 问题分析

### 当前行为（错误）

```
cyq_perf 接口（reverse_date_range + is_date_anchor=true）

用户命令: python app4/main.py --interface cyq_perf --start_date 20260205 --end_date 20260220

执行流程:
1. params_builder._detect_scenario()
   - is_stock_loop = False (mode = reverse_date_range)
   - 返回 DownloadScenario.DIRECT
   
2. pagination.compose()
   - is_date_anchor_interface = True
   - 跳过 time_range 处理
   - 但参数仍然是 {start_date: "20260205", end_date: "20260220"}
   
3. API 收到: {"start_date": "20260205", "end_date": "20260220"}
   - 报错: "参数校验失败, ts_code,trade_date至少输入一个参数"
```

### 预期行为（正确）

```
cyq_perf 接口（reverse_date_range + is_date_anchor=true）

用户命令: python app4/main.py --interface cyq_perf --start_date 20260205 --end_date 20260220

执行流程:
1. params_builder._detect_scenario()
   - 检测到 is_date_anchor=true 且 mode=reverse_date_range
   - 返回 DownloadScenario.DATE_ANCHOR_RANGE（新场景）
   
2. params_builder._build_date_anchor_range_params_list()
   - 生成多个请求: {trade_date: "20260205"}, {trade_date: "20260206"}, ...
   - 不包含 ts_code 参数
   
3. API 收到多个请求，每个请求格式正确
   - 成功返回数据
```

## 关键设计决策

### 为什么必须新增场景，不能复用 STOCK_LOOP_DATE_ANCHOR？

| 场景 | 参数格式 | API 行为 |
|------|---------|---------|
| `STOCK_LOOP_DATE_ANCHOR` | `{ts_code: "000001.SZ", trade_date: "20260205"}` | 返回指定股票数据 |
| `DATE_ANCHOR_RANGE`（新） | `{trade_date: "20260205"}` | 返回该日所有股票数据 |

**结论**：两者语义不同，参数格式不同，必须分开处理。

### 为什么 pagination.py 无需修改？

`params_builder.py` 已经正确生成了参数列表，`pagination.py` 的 `compose()` 方法只需透传即可。现有的日期锚定处理逻辑（`_apply_date_anchor`）用于处理带 `_start_date/_end_date` 内部参数的情况，而新场景直接生成完整的参数列表，无需额外转换。

### 修改点 4: params_builder.py - 新增参数构建方法

```python
def _build_date_anchor_range_params(
    self,
    start_date: str,
    end_date: Optional[str],
    ts_code: Optional[str]
) -> BuildResult:
    """
    构建日期锚定范围参数
    
    专用于 reverse_date_range + is_date_anchor 的接口（如 cyq_perf）
    将用户提供的日期范围存储为内部参数，后续转换为单个日期锚定值
    
    与 STOCK_LOOP_DATE_ANCHOR 的区别：
    - 不设置 requires_stock_loop=True
    - 不遍历股票列表，只按日期遍历
    """
    params: Dict[str, Any] = {}
    date_anchor_param = self._find_date_anchor_param()
    
    # 使用内部参数存储日期范围
    if start_date:
        params['_date_anchor_start'] = start_date
    if end_date:
        params['_date_anchor_end'] = end_date
    if ts_code:
        params['ts_code'] = ts_code
    
    return BuildResult(
        params=params,
        scenario=DownloadScenario.DATE_ANCHOR_RANGE,
        requires_stock_loop=False,  # 关键：不需要股票循环
        date_anchor_param=date_anchor_param
    )
```

### 修改点 6: params_builder.py - 新增参数列表构建方法

**文件**: `app4/core/params_builder.py`

**位置**: 在 `_build_pro_bar_params_list` 方法后（约第 345 行后）

```python
def _build_date_anchor_range_params_list(self, result: BuildResult) -> List[Dict[str, Any]]:
    """
    构建日期锚定范围参数列表
    
    与 _build_stock_loop_anchor_params_list 的关键区别：
    - 不遍历股票列表
    - 每个参数只包含日期锚定值，不包含 ts_code
    
    Args:
        result: 构建结果
        
    Returns:
        参数列表，如 [{'trade_date': '20260205'}, {'trade_date': '20260206'}, ...]
    """
    start_date = result.params.get('_date_anchor_start')
    end_date = result.params.get('_date_anchor_end')
    date_anchor_param = result.date_anchor_param
    
    if not start_date or not date_anchor_param:
        return []
    
    # 生成每日日期锚定值
    anchor_values = self._generate_daily_anchor_values(start_date, end_date)
    
    params_list = []
    for anchor_value in anchor_values:
        p = {date_anchor_param: anchor_value}
        # 如果用户指定了 ts_code，也传递（用于查询特定股票）
        if result.params.get('ts_code'):
            p['ts_code'] = result.params['ts_code']
        params_list.append(p)
    
    return params_list

def _generate_daily_anchor_values(self, start_date: str, end_date: Optional[str]) -> List[str]:
    """
    生成每日日期锚定值
    
    与 _generate_date_anchor_values 的区别：
    - 后者生成季度末日期（用于 ann_date, end_date 等）
    - 本方法生成每日日期（用于 trade_date）
    
    Args:
        start_date: 开始日期 YYYYMMDD
        end_date: 结束日期 YYYYMMDD
        
    Returns:
        日期字符串列表，如 ['20260205', '20260206', '20260207']
    """
    if not start_date:
        return []
    
    try:
        start = datetime.strptime(start_date, '%Y%m%d')
        end = datetime.strptime(end_date, '%Y%m%d') if end_date else datetime.now()
    except ValueError:
        return []
    
    dates = []
    current = start
    while current <= end:
        dates.append(current.strftime('%Y%m%d'))
        current += timedelta(days=1)
    
    return dates
```

**注意**: 确保文件顶部已导入 `timedelta`:
```python

```

## 为什么不需要修改 pagination.py

原有方案提议在 `pagination.py` 中新增 `_apply_date_anchor_range` 方法，经分析后发现**不必要**：

1. **参数构建已完成**：`params_builder.py` 的新方法已正确生成完整的参数列表
2. **现有逻辑兼容**：`pagination.py` 的 `compose()` 方法会直接透传参数列表
3. **避免重复处理**：若在 `pagination.py` 再次处理，会导致重复转换

### 现有 compose() 方法行为

```python
# pagination.py 第 65-112 行
def compose(self, base_params: Dict[str, Any]) -> Iterator[Dict[str, Any]]:
    params_stream = [base_params]
    
    # 对于 DATE_ANCHOR_RANGE 场景，参数列表已由 params_builder 生成
    # 直接传入 execute() 方法，不会进入以下维度处理：
    
    if self._is_enabled("time_range") and not is_date_anchor_interface:
        params_stream = list(self._apply_time_range(params_stream))
    
    if self._is_enabled("stock_loop"):
        params_stream = list(self._apply_stock_loop(params_stream))
    
    # ...
    
    yield from params_stream
```
