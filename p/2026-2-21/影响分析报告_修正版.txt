修改目的
--------
解决 cyq_perf 接口在配置了 is_date_anchor: true 后，使用 --start_date 和 --end_date 
参数时仍然传递范围参数而非单个日期参数的问题。

------------
1. app4/core/params_builder.py - 场景检测和参数构建逻辑
2. app4/core/pagination.py - 日期锚定维度处理
3. app4/update/update_manager.py - 更新模式参数处理
4. 其他文件修改：dedup.py, schema_manager.py, storage.py（辅助性修改）

5. disclosure_date (stock_loop模式, is_date_anchor=true)
   - 影响：用户指定日期范围时，会按季度生成多个请求

================================================================================
修正说明：实际执行路径分析
================================================================================

原报告第91-92行存在不准确描述：
"对于非 stock_loop 模式（如 reverse_date_range）的日期锚定接口会使用 stock_loop 流程处理"

【修正】实际执行路径分析：

1. 更新模式（--update）：
   - 调用路径：update_manager._execute_download() -> pagination_executor.execute()
   - 在 pagination.compose() 方法中，日期锚定接口走特殊路径（第91-94行）：
     ```python
     if is_date_anchor_interface:
         params_stream = list(self._apply_date_anchor(params_stream))
         return  # 直接返回，不进入 stock_loop 维度
     ```
   - 结论：**日期锚定接口在更新模式下是日期单循环，不涉及股票循环**

2. 普通下载模式：
   - 调用路径：main.py -> params_builder -> build_params_list()
   - 对于日期锚定接口，返回 STOCK_LOOP_DATE_ANCHOR 场景
   - _build_stock_loop_anchor_params_list() 会生成 (stock × date) 组合参数
   - 结论：**普通下载模式下会生成股票×日期双循环参数列表**

3. cyq_perf 实际行为验证（从运行日志）：
   - 每次请求返回约 5465 条记录（接近股票总数 5484）
   - 说明每次请求不带 ts_code 参数，按日期查询返回所有股票数据
   - 实际执行的是日期单循环

四、代码逻辑不一致问题
---------------------

【发现】params_builder 和 pagination_executor 的处理逻辑不一致：

| 组件 | 更新模式行为 | 普通模式行为 |
|------|-------------|-------------|
| pagination_executor | 日期单循环 | 同左 |
| params_builder | 不适用 | 股票×日期双循环 |

对于 cyq_perf 这类 reverse_date_range + is_date_anchor 的接口：
- 更新模式：日期单循环（正确，每次返回全部股票）
- 普通模式：股票×日期双循环（可能产生大量冗余请求）

建议：
1. 统一两套逻辑，或在 params_builder 中增加判断：
   - 若 API 支持不传 ts_code 返回全部股票，则使用日期单循环
   - 若 API 必须传 ts_code，才使用股票×日期双循环

2. 简化冗余的 if/else 分支（第109-117行）

================================================================================
总结
================================================================================

1. 正面影响：成功修复了 cyq_perf 接口在更新模式下的参数传递问题

2. 潜在风险：
   - 普通下载模式下可能产生不必要的股票循环请求
   - 需要统一 params_builder 和 pagination_executor 的处理逻辑

3. 代码质量问题：
   - 存在一处冗余的 if/else 分支判断
   - 两套参数构建逻辑存在不一致

4. 建议后续优化：
   - 简化冗余代码
   - 统一日期锚定接口的处理路径
