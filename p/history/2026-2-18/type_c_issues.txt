一、问题概述
--------------------------------------------------------------------------------

测试脚本: /home/quan/testdata/aspipe_v4/p/interface5/test_type_abcd.sh
输出目录: /home/quan/testdata/aspipe_v4/p/interface5/output

发现 Type C 接口（日期锚定模式）存在以下问题：
1. 任务数量过多
2. 下载范围超出用户指定
3. 可能存在重复任务


二、Type C 接口列表
--------------------------------------------------------------------------------

| 接口名称        | 锚定参数    | is_date_anchor | start_date 参数 |
|----------------|-------------|----------------|-----------------|
| top10_holders  | period      | true           | 无              |
| stk_rewards    | end_date    | true           | 无              |
| disclosure_date| end_date    | true           | 无              |
| dividend       | ann_date    | true           | 无              |
| pledge_stat    | end_date    | true           | 无              |

注意：所有 Type C 接口的 YAML 配置文件都没有定义 start_date 参数。


三、问题详情
--------------------------------------------------------------------------------

【问题1】任务数量过多
--------------------------------------------------------------------------------

现象：
  用户指定 --start_date 20200101 --end_date 20241231
  实际生成了 63+ 个下载任务，而非预期的 ~10 个

代码位置：
  app4/core/downloader.py 第 478 行

  start_date = stock_params.get('start_date', DEFAULT_STOCK_START_DATE)
  
  DEFAULT_STOCK_START_DATE = "20050101"  (定义在 constants.py)

原因：
  Type C 接口没有 start_date 参数定义，stock_params.get('start_date') 返回 None
  代码使用默认值 20050101，导致从 2005 年开始生成锚点值


【问题2】下载范围超出用户指定
--------------------------------------------------------------------------------

现象：
  用户指定范围: 20200101 ~ 20241231
  实际使用范围: 20050101 ~ 20241231

代码位置：
  app4/core/coverage_manager.py 第 1238-1247 行

  def _generate_report_periods(self, start_date: str, end_date: str) -> List[str]:
      periods = []
      start_year = int(start_date[:4])  # 使用错误的 start_date
      end_year = int(end_date[:4])
      quarter_ends = ["0331", "0630", "0930", "1231"]

      for year in range(start_year - 1, end_year + 2):
          for qe in quarter_ends:
              period = f"{year}{qe}"
              if start_date <= period <= end_date:
                  periods.append(period)
      return sorted(periods)

原因：
  传入的 start_date 是默认值 20050101，而非用户指定的 20200101


【问题3】日期参数传递问题
--------------------------------------------------------------------------------

代码位置：
  app4/main.py 第 681-684 行

  user_provided_start_date = '--start_date' in sys.argv
  user_provided_end_date = '--end_date' in sys.argv
  user_provided_dates = user_provided_start_date or user_provided_end_date

问题：
  虽然检测到用户提供了日期参数，但对于 Type C 接口：
  - YAML 配置没有 start_date 参数定义
  - 命令行传入的 --start_date 无法正确传递到缺口检测逻辑


四、问题链条
--------------------------------------------------------------------------------

1. 用户命令行: --start_date 20200101 --end_date 20241231

2. main.py 检测到 user_provided_dates = True

3. 但 Type C 接口的 YAML 没有 start_date 参数定义

4. downloader.py 第 478 行:
   stock_params.get('start_date') 返回 None

5. 使用默认值 DEFAULT_STOCK_START_DATE = "20050101"

6. coverage_manager.py 用 20050101 生成锚点值

7. 最终下载范围远超用户预期

  - top10_holders: 有问题

六、建议修复方案
--------------------------------------------------------------------------------

方案1：在 Type C 接口的 YAML 配置中添加 start_date 参数定义

  parameters:
    start_date:
      description: 开始日期 YYYYMMDD
      required: false
      type: string
    end_date:
      is_date_anchor: true
      ...

方案2：修改 downloader.py，对 Type C 接口特殊处理

  在获取 start_date 前，检查接口类型，如果是 date_anchor 模式，
  应该从命令行参数或 context 中获取用户指定的日期

方案3：修改 coverage_manager.py 的 _detect_date_anchor_gaps 方法

  增加 user_provided_dates 参数的判断逻辑，
  当用户提供了日期时，使用用户指定的范围而非默认值
