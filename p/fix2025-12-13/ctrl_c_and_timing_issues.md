# stk_factor_pro API Ctrl+C中断和长时间间隔问题诊断与修复方案

## 问题概述

在分析 `stk_factor_pro` 接口使用过程中发现以下两个关键问题：

1. **Ctrl+C 中断无法停止程序** - 程序无法正常响应中断信号
2. **API调用存在长时间间隔** - 两次API调用间有数分钟的延迟

## 问题详细分析

### 1. Ctrl+C中断处理问题

**问题现象**：
- 日志显示：需要多次按下Ctrl+C才能停止程序
- 最终显示“Killed”，表明进程被强制终止而非优雅退出
- `^C`信号后，程序仍然继续执行API调用（如16:07:40仍成功调用API）

**根本原因**：
`/home/quan/testdata/aspipe_v4/app/date_range_downloader.py` 中使用了 `concurrent.futures.ThreadPoolExecutor` 进行并发下载：

```python
with ThreadPoolExecutor(max_workers=max_workers) as executor:
    # 线程池中的任务无法接收主线程的中断信号
```

- 主线程的中断信号不会自动传播到工作线程
- `download_func()` 调用可能是长时间阻塞的网络请求，在此期间无法响应中断
- 即使主线程检查到中断标志，线程池中的任务仍在继续执行

### 2. 长时间间隔问题

**问题现象**：
- 日志显示：16:00:50 到 16:06:14，间隔约5分24秒
- 远超过按30次/分钟限制的理论等待时间（2秒）

**根本原因**：
- **服务端更严格的限频**：TuShare服务端可能对连续请求有比文档更严格的限制
- **多线程速率控制不当**：各线程可能独立计算速率限制，导致整体控制不协调
- **API重试机制**：错误重试时的指数退避机制导致额外等待
- **网络超时**：API调用可能因网络问题而超时，导致长时间阻塞

## 修复方案

### 1. 改进中断处理机制

**修复位置**：`date_range_downloader.py`

**具体措施**：
1. **使用可取消的任务**：为线程池任务添加中断检查机制
2. **全局中断标志**：使用线程安全的中断标志在主线程和工作线程间共享
3. **优雅关闭线程池**：在中断时正确关闭并等待线程池

```python
# 添加线程安全的中断控制
import threading
interrupted_event = threading.Event()

# 在每个下载任务中定期检查中断
def download_task_with_interrupt_check():
    if interrupted_event.is_set():
        return  # 退出任务
    # 执行下载
```

### 2. 为API调用添加超时机制

**修复位置**：`tushare_api.py` 和各接口模块

**具体措施**：
1. **设置网络超时**：为tushare API调用添加合理的超时时间
2. **中断可取消的请求**：使用支持中断的网络库或机制

### 3. 改进速率限制算法

**修复位置**：`tushare_api.py` 的 `_advanced_rate_limit` 方法

**具体措施**：
1. **中心化的速率控制**：使用全局的API调用计数器
2. **更精确的时间控制**：优化时间间隔计算，避免过度保守的等待

### 4. 优化线程池管理

**修复位置**：`date_range_downloader.py`

**具体措施**：
1. **中断时取消任务**：使用`executor.shutdown(wait=False)`快速关闭
2. **任务状态监控**：在主循环中检查任务状态，及时响应中断

## 实施建议

### 高优先级
1. 实现线程安全的中断机制
2. 为API调用添加超时控制
3. 改进线程池的中断处理

### 中优先级
1. 优化速率限制算法
2. 更新全局中断标志检查机制

### 低优先级
1. 添加更详细的中断日志
2. 优化并发策略

## 验证步骤

1. 运行下载任务，测试Ctrl+C是否能立即中断
2. 检查API调用间隔是否符合预期
3. 验证中断后是否无残留进程
4. 确认数据下载完整性未受影响

## 结论

当前问题的根源在于并发控制和中断处理机制不当。修复需要从架构层面解决多线程环境下的信号处理问题，确保中断信号能正确传递并停止所有正在运行的下载任务。
