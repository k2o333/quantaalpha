# 6个接口参数不匹配问题解决方案

## 问题概述

以下6个接口存在**参数定义与代码强制传递不匹配**的问题，导致查询失败或返回空结果：

| 接口 | 配置定义 | 代码强制传递 | 结果 |
|------|---------|-------------|------|
| **disclosure_date** | 无 `start_date`，`end_date`=财报周期 | `start_date` + `end_date` | 查询未来日期，返回0条 |
| **dividend** | 无 `start_date`，无 `end_date` | `start_date` + `end_date` | 参数被忽略，返回0条 |
| **pledge_detail** | 只有 `ts_code` | `start_date` + `end_date` | 参数被忽略 |
| **pledge_stat** | 无 `start_date`，`end_date`=截止日期 | `start_date` + `end_date` | 查询未来日期，返回0条 |
| **stk_rewards** | 无 `start_date`，`end_date`=报告期 | `start_date` + `end_date` | 查询未来日期，返回0条 |
| **top10_holders** | 无 `start_date`，使用 `period` | `start_date` + `end_date` | 使用默认period，参数被忽略 |

## 根本原因

**代码逻辑**（`app4/main.py` 第608-610行）：
```python
params = {
    'start_date': args.start_date,  # 强制传递！
    'end_date': args.end_date       # 强制传递！
}
```

这段代码对所有接口统一传递 `start_date` 和 `end_date`，但这些接口的配置文件并未定义这些参数，或定义了不同的含义。

## 解决方案

### 方案一：根据接口配置动态构建参数（推荐）

**修改位置**：`app4/main.py` 第608-615行

**修改前**：
```python
params = {
    'start_date': args.start_date,
    'end_date': args.end_date
}

# 如果指定了股票代码，添加到参数中
if args.ts_code:
    params['ts_code'] = args.ts_code
```

**修改后**：
```python
# 根据接口配置动态构建参数，而不是强制传递 start_date/end_date
parameter_config = interface_config.get('parameters', {})
params = {}

# 只传递接口定义的参数
if 'ts_code' in parameter_config and args.ts_code:
    params['ts_code'] = args.ts_code

# 对于有 start_date 定义的接口，传递日期参数
if 'start_date' in parameter_config:
    params['start_date'] = args.start_date
    if 'end_date' in parameter_config:
        params['end_date'] = args.end_date
    else:
        # 如果只有 start_date 没有 end_date，尝试传递 end_date 看看接口是否接受
        params['end_date'] = args.end_date
else:
    # 没有 start_date 定义的接口（如 disclosure_date, dividend 等）
    # 这些接口通常只接受特定参数，不传递日期范围
    logger.debug(f"Interface {interface_name} does not accept start_date/end_date, skipping date params")
```

### 方案二：特殊接口白名单处理

**修改位置**：`app4/main.py` 第608-615行

**修改后**：
```python
# 定义不接受标准日期范围的接口
NO_DATE_RANGE_INTERFACES = [
    'disclosure_date', 'dividend', 'pledge_detail', 
    'pledge_stat', 'stk_rewards', 'top10_holders'
]

if interface_name in NO_DATE_RANGE_INTERFACES:
    # 这些接口只传递 ts_code，让 TuShare 返回该股票的所有历史数据
    params = {}
    if args.ts_code:
        params['ts_code'] = args.ts_code
    logger.info(f"Interface {interface_name} does not use date range params, fetching all history for ts_code")
else:
    # 标准接口传递日期范围
    params = {
        'start_date': args.start_date,
        'end_date': args.end_date
    }
    if args.ts_code:
        params['ts_code'] = args.ts_code
```

### 方案三：分页模式判断（最简洁）

**修改位置**：`app4/main.py` 第608-615行及后续 stock_loop 处理逻辑

**核心逻辑**：对于 `stock_loop` 模式的接口，在循环内部处理单只股票时，应该只传递 `ts_code`：

```python
# 在 stock_loop 模式下，修改参数传递逻辑
if pagination_config.get('mode') == 'stock_loop':
    # stock_loop 模式下，外层不传递日期参数
    # 日期参数会在 _download_single_stock 方法中根据股票上市日期自动设置
    params = {}
    if args.ts_code:
        params['ts_code'] = args.ts_code
else:
    # 非 stock_loop 模式，传递标准日期参数
    params = {
        'start_date': args.start_date,
        'end_date': args.end_date
    }
    if args.ts_code:
        params['ts_code'] = args.ts_code
```

然后修改 `downloader.py` 中的 `_download_single_stock` 方法：

```python
# 修改前（第442-448行）
if 'start_date' not in stock_params:
    list_date = stock.get('list_date', '20050101')
    stock_params['start_date'] = list_date
if 'end_date' not in stock_params:
    stock_params['end_date'] = datetime.now().strftime('%Y%m%d')

# 修改后 - 检查接口是否接受这些参数
parameter_config = interface_config.get('parameters', {})
if 'start_date' in parameter_config:
    if 'start_date' not in stock_params:
        list_date = stock.get('list_date', '20050101')
        stock_params['start_date'] = list_date
    if 'end_date' not in stock_params:
        stock_params['end_date'] = datetime.now().strftime('%Y%m%d')
else:
    # 接口不接受 start_date，移除这些参数（如果之前被添加）
    stock_params.pop('start_date', None)
    stock_params.pop('end_date', None)
```

## 推荐方案：方案三（分页模式判断）

**理由**：
1. **逻辑清晰**：根据分页模式决定参数传递策略
2. **可扩展性好**：新增 stock_loop 接口时自动适用
3. **代码改动小**：只需修改参数构建逻辑，无需维护白名单
4. **符合设计初衷**：stock_loop 模式本来就是为单股票全历史查询设计的

## 风险控制

### 潜在风险

1. **数据量增加**：不再按日期过滤，可能下载更多历史数据
2. **API 调用次数**：单只股票可能需要多次调用（分页）

### 缓解措施

1. **覆盖率管理**：利用 `coverage_manager` 跳过已存在的股票
2. **增量模式**：支持 `--incremental` 只下载缺失数据
3. **用户控制**：用户仍可通过 `--ts_code` 指定单只股票
