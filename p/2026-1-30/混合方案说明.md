# 混合方案说明

## 问题背景

以下6个接口存在**参数定义与代码强制传递不匹配**的问题，导致查询失败或返回空结果：

| 接口 | 配置定义 | 代码强制传递 | 结果 |
|------|---------|-------------|------|
| **disclosure_date** | 无 `start_date`，`end_date`=财报周期 | `start_date` + `end_date` | 查询未来日期，返回0条 |
| **dividend** | 无 `start_date`，无 `end_date` | `start_date` + `end_date` | 参数被忽略，返回0条 |
| **pledge_detail** | 只有 `ts_code` | `start_date` + `end_date` | 参数被忽略 |
| **pledge_stat** | 无 `start_date`，`end_date`=截止日期 | `start_date` + `end_date` | 查询未来日期，返回0条 |
| **stk_rewards** | 无 `start_date`，`end_date`=报告期 | `start_date` + `end_date` | 查询未来日期，返回0条 |
| **top10_holders** | 无 `start_date`，使用 `period` | `start_date` + `end_date` | 使用默认period，参数被忽略 |

## 根本原因

代码在以下位置强制添加 `start_date` 和 `end_date`，而不考虑接口实际支持的参数：

1. **app4/main.py:608-610** - 命令行参数处理
2. **app4/core/downloader.py:442-448** - `download_single_stock()` 方法
3. **app4/core/pagination.py:188-193** - `generate_stock_params()` 方法

## 单一方案的局限性

### 方案1：根据接口配置动态构建参数

**位置**：`main.py`

```python
parameter_config = interface_config.get('parameters', {})
params = {}

if 'ts_code' in parameter_config and args.ts_code:
    params['ts_code'] = args.ts_code

if 'start_date' in parameter_config:
    params['start_date'] = args.start_date
```

**问题**：
- disclosure_date 配置中没有 `start_date`，但有 `end_date`（财报周期）
- 按此逻辑不会传递任何日期参数
- 但 disclosure_date 需要 `end_date` 作为财报周期

### 方案2：特殊接口白名单处理

**位置**：`main.py`

```python
NO_DATE_RANGE_INTERFACES = ['disclosure_date', 'dividend', ...]
if interface_name in NO_DATE_RANGE_INTERFACES:
    params = {}
```

**问题**：
- 需要维护硬编码的白名单
- 违反"配置驱动"的设计原则
- 新增接口需要手动添加

### 方案3：分页模式判断

**位置**：`main.py`

```python
if is_stock_loop:
    params = {}  # 不传日期参数
else:
    params = {'start_date': ..., 'end_date': ...}
```

**问题**：
- 只解决了 main.py 层面的问题
- **未处理内部参数需求**：downloader.py 仍然会强制添加不支持的参数
- 例如 disclosure_date 不支持 `start_date`，但 downloader.py 会添加

### 方案4：参数过滤

**位置**：`downloader.py`

```python
supported_params = interface_config.get('parameters', {})
stock_params = {k: v for k, v in stock_params.items() if k in supported_params}
```

**问题**：
- 与方案1类似，无法区分"不支持日期参数"和"支持特定日期参数"的场景

### 方案5：修改接口配置

**位置**：yaml 配置文件

**问题**：
- 会改变接口的语义！
- disclosure_date 的 `end_date` 是"财报周期"，不是查询结束日期
- TuShare API 可能不支持这些参数

## 混合方案

### 设计思路

混合方案结合**分页模式判断**和**参数配置过滤**，在**两个层面**精确控制参数：

1. **层面1：main.py** - 根据 stock_loop 模式决定是否传递日期参数
2. **层面2：downloader.py + pagination.py** - 根据接口配置精确设置参数

### 实现代码

#### 修改位置1：app4/main.py

**位置**：第608-615行（参数构建逻辑）

```python
# 检查分页模式
pagination_config = interface_config.get('pagination', {})
is_stock_loop = (
    pagination_config.get('enabled', False) and
    pagination_config.get('mode') == 'stock_loop'
)

if is_stock_loop:
    # stock_loop 模式：不传递日期参数，让接口返回全历史
    params = {}
    if args.ts_code:
        params['ts_code'] = args.ts_code
    logger.info(f"Using stock_loop mode for {interface_name}, fetching full history")
else:
    # 其他模式：传递标准日期参数
    params = {
        'start_date': args.start_date,
        'end_date': args.end_date
    }
    if args.ts_code:
        params['ts_code'] = args.ts_code
```

#### 修改位置2：app4/core/downloader.py

**位置**：`download_single_stock()` 方法（约第442-448行）

```python
stock_params = params.copy()
stock_params['ts_code'] = stock['ts_code']

# 根据接口配置决定是否设置日期参数
parameter_config = interface_config.get('parameters', {})

# 只设置接口支持的参数
if 'start_date' in parameter_config and 'start_date' not in stock_params:
    list_date = stock.get('list_date', '20050101')
    stock_params['start_date'] = list_date

if 'end_date' in parameter_config and 'end_date' not in stock_params:
    from datetime import datetime
    stock_params['end_date'] = datetime.now().strftime('%Y%m%d')

# 对于不支持日期参数的接口，移除被意外添加的日期参数
if 'start_date' not in parameter_config:
    stock_params.pop('start_date', None)
if 'end_date' not in parameter_config:
    stock_params.pop('end_date', None)

logger.info(f"Downloading data for stock {stock['ts_code']}, params: {stock_params}")
```

#### 修改位置3：app4/core/pagination.py

**位置**：`generate_stock_params()` 方法（约第188-193行）

```python
stock_params = base_params.copy()
stock_params['ts_code'] = ts_code

# 检查接口配置
parameter_config = self.context.interface_config.get('parameters', {})

# 只设置接口支持的参数
if 'start_date' in parameter_config and 'start_date' not in stock_params:
    list_date = stock.get('list_date', '20050101')
    stock_params['start_date'] = list_date

if 'end_date' in parameter_config and 'end_date' not in stock_params:
    stock_params['end_date'] = datetime.now().strftime('%Y%m%d')

# 移除不支持的参数
if 'start_date' not in parameter_config:
    stock_params.pop('start_date', None)
if 'end_date' not in parameter_config:
    stock_params.pop('end_date', None)

yield stock_params, stock
```

## 混合方案的优势

### 1. 在两个层面精确控制

| 层面 | 作用 | 修改前 | 修改后 |
|------|------|--------|--------|
| **main.py** | 入口参数构建 | 强制传递 `start_date` + `end_date` | 根据 stock_loop 模式决定 |
| **downloader.py** | 内部参数处理 | 强制添加日期参数 | 根据接口配置过滤 |
| **pagination.py** | 参数生成器 | 强制添加日期参数 | 根据接口配置过滤 |

### 2. 符合架构设计

- **stock_loop 模式**：本来就是为单股票全历史查询设计的，不需要外部日期范围
- **配置驱动**：参数由接口配置决定，不是硬编码
- **职责分离**：main.py 负责入口控制，downloader.py 负责内部逻辑

### 3. 正确处理各种场景

| 接口 | 配置定义 | 混合方案最终参数 | 结果 |
|------|---------|----------------|------|
| **disclosure_date** | `ts_code`, `end_date`(财报周期) | `ts_code`, `end_date` | ✅ 正确 |
| **dividend** | `ts_code` | `ts_code` | ✅ 正确 |
| **pledge_detail** | `ts_code` | `ts_code` | ✅ 正确 |
| **pledge_stat** | `ts_code`, `end_date`(截止日期) | `ts_code`, `end_date` | ✅ 正确 |
| **stk_rewards** | `ts_code`, `end_date`(报告期) | `ts_code`, `end_date` | ✅ 正确 |
| **top10_holders** | `ts_code`, `period` | `ts_code`, `period` | ✅ 正确 |

### 4. 可扩展性好

- 新增 stock_loop 接口时自动适用
- 无需维护白名单
- 配置驱动，无需修改代码

## 解决的问题

### 问题1：强制传递不支持的参数

**修复前**：
```python
# main.py
params = {
    'start_date': '20230101',
    'end_date': '20260131',
    'ts_code': '000001.SZ'
}

# downloader.py
stock_params['start_date'] = '19900101'  # 强制添加！
stock_params['end_date'] = '20260131'    # 强制添加！

# 最终传递给 API
{
    'ts_code': '000001.SZ',
    'start_date': '19900101',  # ❌ disclosure_date 不支持
    'end_date': '20260131'     # ❌ 语义错误（应该是财报周期）
}
```

**修复后**：
```python
# main.py (stock_loop 模式)
params = {'ts_code': '000001.SZ'}

# downloader.py
parameter_config = interface_config.get('parameters', {})
# disclosure_date 没有 start_date，有 end_date
if 'start_date' not in parameter_config:
    stock_params.pop('start_date', None)  # ✅ 移除
if 'end_date' in parameter_config:
    stock_params['end_date'] = datetime.now().strftime('%Y%m%d')  # ✅ 设置

# 最终传递给 API
{
    'ts_code': '000001.SZ',
    'end_date': '20260131'  # ✅ 正确（作为财报周期）
}
```

### 问题2：接口参数语义混淆

**修复前**：
- 所有接口的 `end_date` 都被当作"查询结束日期"
- 但 disclosure_date 的 `end_date` 是"财报周期"
- pledge_stat 的 `end_date` 是"截止日期"
- stk_rewards 的 `end_date` 是"报告期"

**修复后**：
- 根据接口配置决定是否传递 `end_date`
- 不传递接口不支持的参数
- 保留接口支持的参数及其正确语义

### 问题3：硬编码维护成本

**修复前**：
```python
# 需要维护白名单
NO_DATE_RANGE_INTERFACES = [
    'disclosure_date', 'dividend', 'pledge_detail',
    'pledge_stat', 'stk_rewards', 'top10_holders'
]
```

**修复后**：
```python
# 完全配置驱动，无需硬编码
parameter_config = interface_config.get('parameters', {})
```

## 总结

混合方案通过在**两个层面**精确控制参数传递：

1. **main.py 层面**：根据 stock_loop 模式避免传递不必要的日期参数
2. **downloader.py + pagination.py 层面**：根据接口配置过滤和设置参数

这种方案：
- ✅ 完全解决6个接口参数不匹配问题
- ✅ 符合系统的架构设计（stock_loop 模式语义）
- ✅ 遵循配置驱动原则
- ✅ 无需维护硬编码白名单
- ✅ 可扩展性好，新增接口自动适用
- ✅ 正确处理不同接口的参数语义差异
