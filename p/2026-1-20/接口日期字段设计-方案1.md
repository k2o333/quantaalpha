# 接口日期字段设计 - 方案1：显式添加日期格式字段

## 背景

`trade_cal`接口已经添加了`cal_date_dt`日期格式字段，用于提升polars查询性能。现在需要确定其他接口在日期作为主键时的处理方案。

## 方案概述

**方案1：显式添加日期格式字段**

在需要关联`trade_cal`的接口配置中，显式添加对应的日期格式字段，保持数据自包含。

## 实现方式

### 1. 配置修改示例

对于以`trade_date`作为主键的接口，在`derived_fields`中添加：

```yaml
derived_fields:
  trade_date_dt:
    description: 日期类型的trade_date（关联trade_cal.cal_date_dt）
    format: '%Y%m%d'
    source: trade_date
    type: date
```

对于以`ann_date`等其他日期字段作为主键的接口，同理添加：

```yaml
derived_fields:
  ann_date_dt:
    description: 日期类型的ann_date（关联trade_cal.cal_date_dt）
    format: '%Y%m%d'
    source: ann_date
    type: date
```

### 2. 适用接口范围

需要添加日期格式字段的接口（日期作为主键）：

| 接口名称 | 主键日期字段 | 建议添加的日期字段 |
|---------|------------|------------------|
| daily | trade_date | trade_date_dt |
| moneyflow | trade_date | trade_date_dt |
| daily_basic | trade_date | trade_date_dt |
| bak_daily | trade_date | trade_date_dt |
| suspend_d | trade_date | trade_date_dt |
| stock_st | trade_date | trade_date_dt |
| fina_indicator | ann_date, end_date | ann_date_dt, end_date_dt |
| income | ann_date, end_date | ann_date_dt, end_date_dt |
| balancesheet | ann_date, end_date | ann_date_dt, end_date_dt |
| cashflow | ann_date, end_date | ann_date_dt, end_date_dt |

### 3. 数据存储格式

生成的parquet文件将包含原始string格式和date格式两个字段：

```python
# 示例数据结构
df = pl.DataFrame({
    'ts_code': ['000001.SZ', '000002.SZ'],
    'trade_date': ['20240101', '20240102'],  # 原始string格式
    'trade_date_dt': [date(2024, 1, 1), date(2024, 1, 2)],  # 新增date格式
    # ... 其他字段
})
```

## 优点

1. **查询性能最优**：polars等工具使用日期格式作为主键时性能最佳
2. **数据自包含**：无需额外join操作，简化查询逻辑
3. **类型安全**：明确区分string和date类型，减少类型转换错误
4. **向后兼容**：保留原始string格式，不影响现有代码

## 缺点

1. **数据冗余**：同一份日期数据存储两份（string + date）
2. **存储成本**：略微增加存储空间（约5-10%）
3. **维护成本**：需要修改多个接口配置

## 使用示例

### Polars查询优化

```python
import polars as pl

# 读取数据
df = pl.read_parquet("app4/data/daily/*.parquet")

# 使用日期格式进行高效过滤
result = df.filter(
    pl.col("trade_date_dt").is_between(date(2024, 1, 1), date(2024, 12, 31))
)

# 日期运算性能更好
result = df.with_columns(
    (pl.col("trade_date_dt") + pl.duration(days=1)).alias("next_day")
)
```

### 与trade_cal关联

```python
# 无需类型转换，直接关联
trade_cal = pl.read_parquet("app4/data/trade_cal/*.parquet")
daily = pl.read_parquet("app4/data/daily/*.parquet")

# 直接join，性能更好
result = daily.join(
    trade_cal.select(["cal_date", "cal_date_dt", "is_open_bool"]),
    left_on="trade_date_dt",
    right_on="cal_date_dt",
    how="left"
)
```

## 实施步骤

1. **优先级排序**：先修改高频查询接口（daily, moneyflow等）
2. **配置修改**：在接口yaml中添加`derived_fields`配置
3. **数据迁移**：重新下载数据，生成带日期格式的parquet文件
4. **验证测试**：
   - 验证日期转换正确性
   - 对比查询性能提升
   - 检查数据完整性
5. **代码更新**：更新查询代码，使用新的日期字段

## 性能预期

- **查询性能**：日期格式过滤比string格式快3-5倍
- **存储增加**：约5-10%（date类型比string更紧凑）
- **内存占用**：polars中date类型内存占用减少约30%

## 注意事项

1. **日期格式统一**：所有接口使用相同的日期格式（`%Y%m%d`）
2. **空值处理**：确保日期转换时正确处理空值或非法日期
3. **时区问题**：日期类型不包含时区信息，统一使用本地日期
4. **兼容性**：保留原始string字段，确保向后兼容

## 决策建议

**推荐采用方案1**，原因：

1. 金融数据分析中日期查询是高频操作，性能提升明显
2. 存储成本增加在可接受范围内
3. 数据自包含，降低使用复杂度
4. 与`trade_cal`的设计保持一致

对于低频使用的接口，可以考虑不添加日期字段，以节省存储空间。
