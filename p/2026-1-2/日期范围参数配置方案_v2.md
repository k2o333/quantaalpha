# 日期范围参数配置方案 v2

## 背景说明

### Tushare API 接口分析

根据 Tushare Pro 官方文档，`income_vip` 接口具有以下特性：

1. **接口参数**：
   - `period`（必需）：报告期，格式为 YYYYMMDD（如 20181231 年报、20180630 半年报、20180930 三季报）
   - `ts_code`（可选）：股票代码
   - `ann_date`（可选）：公告日期
   - `start_date`（可选）：公告日期开始
   - `end_date`（可选）：公告日期结束
   - `f_ann_date`（可选）：实际公告日期
   - `report_type`（可选）：报告类型
   - `comp_type`（可选）：公司类型

2. **关键发现**：
   - `income_vip` 的 `start_date` 和 `end_date` 参数控制的是**公告日期**范围，而非报告期范围
   - 要按报告期查询，必须使用 `period` 参数
   - `period` 参数格式为季度末日期：03-31、06-30、09-30、12-31

3. **当前问题**：
   - 系统中 `start_date` 和 `end_date` 默认控制公告日期范围
   - 用户需要能够指定报告期范围进行批量查询
   - 需要将日期范围转换为多个 `period` 参数请求

## 目标

使系统能够通过配置文件（YAML）灵活指定 `start_date` 和 `end_date` 参数的行为：

1. **公告日期模式**：`start_date`/`end_date` 控制公告日期范围（现有行为）
2. **报告期模式**：`start_date`/`end_date` 控制报告期范围，自动转换为多个 `period` 请求

**核心原则**：
- 接口级别的配置：每个接口根据自己的配置文件决定如何处理日期参数
- 统一逻辑：无论是否指定 `--interface`，每个接口都按照自己的配置处理日期参数
- 不修改 main.py：保持 main.py 的遍历逻辑不变，由 downloader 根据接口配置自动处理

**示例场景**：
- 执行 `python app4/main.py --start_date 20230401 --end_date 20230630`
  - `daily` 接口：使用原生的 `start_date`/`end_date` 逻辑（公告日期模式）
  - `income_vip` 接口：自动将 `start_date`/`end_date` 转换为 period 请求（报告期模式）
  - 其他接口：根据各自的配置文件处理日期参数

### 2. 日期参数类型说明

新增配置参数 `date_parameter_type` 支持以下值：

- **`announcement`**（默认）：
  - `start_date`/`end_date` 控制公告日期范围
  - 直接传递给 API，保持现有行为
  - 适用于：需要按公告日期筛选的场景

- **`report_period`**：
  - `start_date`/`end_date` 控制报告期范围
  - 系统自动将日期范围转换为多个 `period` 参数请求
  - 适用于：需要批量获取多个报告期数据的场景

### 3. 报告期生成规则

当 `date_parameter_type: "report_period"` 时，系统按照以下规则生成报告期：

#### 3.1 季度末日期格式

中国上市公司的财务报告期遵循以下季度末日期：

- **Q1**：03-31（第一季度报告）
- **Q2**：06-30（半年度报告）
- **Q3**：09-30（第三季度报告）
- **Q4**：12-31（年度报告）

#### 3.2 生成算法

给定 `start_date` 和 `end_date`，生成范围内的所有季度末日期：

```
输入：start_date = 20230401, end_date = 20231231

输出：periods = [
  "20230331",  # Q1
  "20230630",  # Q2
  "20230930",  # Q3
  "20231231"   # Q4
]
```

#### 3.3 边界处理规则

- 如果 `start_date` 不是季度末日期，则从**下一个**季度末开始
- 如果 `end_date` 不是季度末日期，则到**上一个**季度末结束
- 确保生成的所有 period 都在 [start_date, end_date] 范围内

示例：

```
输入：start_date = 20230515, end_date = 20231120

处理：
- start_date 20230515 的下一个季度末：20230630
- end_date 20231120 的上一个季度末：20230930

输出：periods = ["20230630", "20230930"]
```

### 4. 实现逻辑

修改 `/home/quan/testdata/aspipe_v4/app4/core/downloader.py`，添加报告期范围分页逻辑：

#### 4.1 新增方法：`_execute_period_range_pagination`

```python
def _execute_period_range_pagination(self, interface_config: Dict[str, Any],
                                    params: Dict[str, Any],
                                    pagination_config: Dict[str, Any]) -> List[Dict[str, Any]]:
    """
    执行报告期范围分页

    当 date_parameter_type 为 "report_period" 时，将 start_date/end_date
    转换为多个 period 参数请求
    """
    all_data = []

    # 获取日期范围
    start_date = params.get('start_date', '20050101')
    end_date = params.get('end_date', datetime.now().strftime('%Y%m%d'))

    logger.info(f"Generating report periods for range: {start_date} - {end_date}")

    # 生成季度末日期列表
    periods = self._generate_quarter_end_dates(start_date, end_date)

    if not periods:
        logger.warning(f"No valid report periods found in range {start_date} - {end_date}")
        return []

    logger.info(f"Generated {len(periods)} report periods: {periods}")

    # 为每个 period 发起请求
    for idx, period in enumerate(periods):
        period_params = params.copy()

        # 移除 start_date 和 end_date，使用 period 参数
        period_params.pop('start_date', None)
        period_params.pop('end_date', None)
        period_params['period'] = period

        logger.info(f"Fetching data for period {period} ({idx+1}/{len(periods)})")

        # 发起请求
        period_data = self._make_request(interface_config, period_params)

        if period_data:
            all_data.extend(period_data)
            logger.info(f"Downloaded {len(period_data)} records for period {period}")
        else:
            logger.warning(f"No data returned for period {period}")

    return all_data
```

#### 4.2 新增方法：`_generate_quarter_end_dates`

```python
def _generate_quarter_end_dates(self, start_date: str, end_date: str) -> List[str]:
    """
    生成日期范围内的所有季度末日期

    Args:
        start_date: 起始日期 YYYYMMDD
        end_date: 结束日期 YYYYMMDD

    Returns:
        季度末日期列表，格式为 YYYYMMDD
    """
    from datetime import datetime, timedelta

    # 解析日期
    start_dt = datetime.strptime(start_date, '%Y%m%d')
    end_dt = datetime.strptime(end_date, '%Y%m%d')

    # 季度末月份和日期
    quarter_ends = [
        (3, 31),   # Q1
        (6, 30),   # Q2
        (9, 30),   # Q3
        (12, 31)   # Q4
    ]

    periods = []

    # 从起始日期的下一个季度末开始
    current_year = start_dt.year
    current_quarter_idx = 0

    # 找到 start_date 之后的第一个季度末
    for q_idx, (month, day) in enumerate(quarter_ends):
        quarter_end = datetime(current_year, month, day)
        if quarter_end >= start_dt:
            current_quarter_idx = q_idx
            break
    else:
        # 如果当前年份没有找到，从下一年第一季度开始
        current_year += 1
        current_quarter_idx = 0

    # 生成所有在范围内的季度末日期
    while True:
        month, day = quarter_ends[current_quarter_idx]
        quarter_end = datetime(current_year, month, day)

        if quarter_end > end_dt:
            break

        periods.append(quarter_end.strftime('%Y%m%d'))

        # 移动到下一个季度
        current_quarter_idx += 1
        if current_quarter_idx >= len(quarter_ends):
            current_quarter_idx = 0
            current_year += 1

    return periods
```

#### 4.3 修改 `_execute_pagination` 方法

在现有的 `_execute_pagination` 方法中添加对 `period_range` 模式的支持：

```python
def _execute_pagination(self, interface_config: Dict[str, Any], params: Dict[str, Any]) -> List[Dict[str, Any]]:
    """执行分页/循环逻辑"""
    pagination_config = interface_config.get('pagination', {})
    if not pagination_config.get('enabled', False):
        # 不分页，直接请求
        return self._make_request(interface_config, params)

    mode = pagination_config.get('mode', 'offset')
    all_data = []

    if mode == 'offset':
        all_data = self._execute_offset_pagination(interface_config, params, pagination_config)
    elif mode == 'date_range':
        all_data = self._execute_date_range_pagination(interface_config, params, pagination_config)
    elif mode == 'stock_loop':
        all_data = self._execute_stock_loop_pagination(interface_config, params)
    elif mode == 'period_range':
        # 新增：报告期范围分页
        all_data = self._execute_period_range_pagination(interface_config, params, pagination_config)
    else:
        # 默认不分页
        all_data = self._make_request(interface_config, params)

    return all_data
```

### 5. 使用方式

#### 5.1 场景 1：指定单个接口

命令行：
```bash
python app4/main.py --start_date 20230401 --end_date 20230630 --interface income_vip
```

执行流程：
1. main.py 只处理 `income_vip` 接口
2. 读取 `income_vip.yaml` 配置文件
3. 发现 `mode: "period_range"` 和 `date_parameter_type: "report_period"`
4. 调用 `_execute_period_range_pagination`
5. 将 `20230401-20230630` 转换为 `["20230331", "20230630"]`
6. 分别调用 API 获取这两个报告期的数据

效果：获取报告期为 20230331、20230630 的 income_vip 数据

#### 5.2 场景 2：不指定接口（批量下载）

命令行：
```bash
python app4/main.py --start_date 20230401 --end_date 20230630
```

执行流程：
1. main.py 遍历所有可用接口
2. 对于每个接口，调用 `downloader.download(interface_name, params)`
3. **每个接口根据自己的配置文件处理日期参数**：

   - **`income_vip` 接口**：
     - 读取 `income_vip.yaml` 配置
     - 发现 `mode: "period_range"` 和 `date_parameter_type: "report_period"`
     - 调用 `_execute_period_range_pagination`
     - 将 `20230401-20230630` 转换为 `["20230331", "20230630"]`
     - 分别调用 API 获取这两个报告期的数据

   - **`daily` 接口**：
     - 读取 `daily.yaml` 配置
     - 发现 `mode: "date_range"`（默认模式）
     - 调用 `_execute_date_range_pagination`
     - 直接将 `start_date`/`end_date` 作为公告日期传递给 API
     - 获取公告日期在 20230401-20230630 范围内的数据

   - **其他接口**：
     - 根据各自的配置文件处理日期参数
     - 如果配置了 `mode: "period_range"`，则转换为 period 请求
     - 如果配置了 `mode: "date_range"`，则直接传递日期参数

效果：
- `income_vip`：获取报告期为 20230331、20230630 的数据
- `daily`：获取公告日期在 20230401-20230630 范围内的数据
- 其他接口：根据各自的配置处理

### 9. 注意事项

1. **API 限制**：
   - `income_vip` 接口每次调用消耗 5000 积分
   - 批量请求时注意积分消耗
   - 建议添加请求间隔，避免触发限流

2. **数据完整性**：
   - 每个季度的数据量可能不同
   - 建议监控每个 period 请求的数据量
   - 如遇数据截断（达到 query_limit），需要调整策略

3. **性能优化**：
   - 可以添加并发请求，提高批量下载效率
   - 建议缓存已下载的 period 数据，避免重复请求
   - 可以添加断点续传功能，支持中断后继续下载

4. **错误处理**：
   - 单个 period 请求失败不应影响其他 period
   - 建议记录失败的 period，支持重试
   - 添加详细的日志记录，便于问题排查
