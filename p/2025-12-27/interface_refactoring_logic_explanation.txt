关于为什么“一个 Client + 一个 Container”能处理复杂接口参数的详细解释

1. 核心原理：底层执行与业务逻辑的剥离
目前的 app 目录中，代码之所以显得乱，是因为它把“接口怎么传参（业务逻辑）”和“如何发送网络请求（基础设施）”混在了一起。重构的核心思想是将这两者彻底解耦。

2. Client 是“动态分发者” (Dynamic Dispatch)
Tushare 的 Python SDK 本身就是高度动态化的。无论接口是 daily 还是 income，最终在 SDK 内部都是通过相同的逻辑发送请求。
- 现状：为 100 个接口写了 100 个包装方法，导致代码量巨大且难以维护。
- 重构：利用 Python 的反射机制（getattr）。Client 只需要一个通用的 download(interface_name, **kwargs) 方法。
  - interface_name 决定调哪个接口。
  - **kwargs 允许透传任何数量、任何名称的参数。
- 结论：Client 变成了一个“哑巴执行者”，它不关心业务，只负责限流、重试和发送请求。

3. 复杂的参数配置去哪了？ (Service & Adapters)
参数的格式化、必填校验、默认值计算等复杂逻辑，不再写在接口类里，而是上移到“业务服务层” (DownloadService) 或“适配器层” (Adapters)。
- Service 层：作为指挥官，它知道下载“日线”需要 start_date，下载“基本面”需要 period。它负责准备好这些参数。
- Adapter 层：如果某个接口的参数格式极其特殊（例如需要将日期对象转为特定字符串），则由专门的 Adapter 负责翻译。
这种结构保证了底层的 Client 永远保持简洁。

4. Container 是“全局大管家” (Dependency Injection)
目前配置混乱的原因是“多头管理”：多个入口、多个文件都在各自读取配置和实例化对象。
- Container 的作用：
  - 统一加载：启动时一次性读取所有配置（Token、积分、代理）。
  - 统一创建：在内存中创建一个配置好的单例 Client 实例。
  - 统一注入：谁需要发请求，Container 就把这个 Client “塞给”谁。
- 结论：这消除了重复创建对象和配置不一致的风险。

5. 总结
- Client 处理的是“共性”：所有的接口都需要限流、报错处理、网络发送。
- Service 处理的是“特性”：每个接口需要什么参数，由业务层决定。
- Container 处理的是“一致性”：确保全局只有一个配置源和一个执行入口。

通过这种方式，原本散落在 app/interfaces/ 下的 12 个模块和数百行重复代码，将被压缩到一个精简、健壮的底层 Client 中。
