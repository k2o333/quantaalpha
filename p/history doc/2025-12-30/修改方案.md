# aspipe_v4 下载频率限制问题修改方案（修订版）

## 问题概述

根据日志分析，系统存在严重的API访问频率限制问题，主要表现为：

- 大量 "访问过于频繁，请合理使用接口，请稍后再试" 错误
- 并发请求过多导致API频率限制
- 缺乏有效的重试和退避策略
- 没有优雅的中断处理

## 解决方案（最小改动版）

### 1. 优化配置文件中的并发设置

修改 `app4/config/settings.yaml`，适度降低并发数：

```yaml
# 并发配置
concurrency:
  max_workers: 6  # 从8降到6，适度降低以减少频率限制
  max_queue_size: 1000
```

        """带重试机制的请求方法"""
        for attempt in range(max_retries + 1):
            try:
                result = self._make_request(interface_config, params)
                
                # 检查是否是频率限制错误
                if self._is_rate_limit_error(result):

                        logger.warning(f"Rate limit hit, waiting {wait_time:.2f}s before retry {attempt + 1}/{max_retries}")
                        time.sleep(wait_time)
                        continue
                    else:
                        logger.error("Rate limit exceeded after all retries")
                        return []
                else:
                    # 如果不是频率限制错误，但有错误信息，也返回空列表
                    if result and isinstance(result, list) and len(result) > 0:
                        # 检查是否包含错误信息
                        first_item = result[0]
                        if isinstance(first_item, dict) and 'error_code' in first_item:
                            logger.error(f"API returned error: {first_item.get('error_msg', 'Unknown error')}")
                            return []
                    return result
                    
            except Exception as e:
                logger.error(f"Request attempt {attempt + 1} failed: {str(e)}")
                if attempt < max_retries:
                    wait_time = (2 ** attempt) + random.uniform(0, 1)
                    time.sleep(wait_time)
                    continue
                else:
                    logger.error(f"All {max_retries + 1} attempts failed due to exception: {str(e)}")
                    return []
        
        return []

    def _is_rate_limit_error(self, result) -> bool:

        if result is None:

                    return '频繁' in error_msg or '访问' in error_msg

                    'error_code': result.get('code'),

                return error_result

            # ... 其余现有代码 ...

### 3. 改进主程序中的中断处理

修改 `app4/main.py`，添加优雅的中断处理：

```python
import signal
import sys

def signal_handler(sig, frame):
    """处理中断信号"""
    print('\n程序被用户中断，正在清理资源...')
    logger.info("Received interrupt signal, shutting down gracefully...")
    # 注意：这里不能直接引用scheduler和storage_manager，因为它们在main函数作用域内
    # 我们需要在main函数内部注册信号处理器
    sys.exit(0)

def main():
    # 注册信号处理器
    signal.signal(signal.SIGINT, signal_handler)
    signal.signal(signal.SIGTERM, signal_handler)
    
    # ... 现有代码 ...
    
    # 在main函数的最后，确保资源被正确清理
    try:
        # ... 现有执行逻辑 ...
    except KeyboardInterrupt:
        logger.info("Keyboard interrupt received, stopping gracefully...")
    finally:
        # 确保资源被清理
        if 'scheduler' in locals():
            scheduler.stop()
        if 'storage_manager' in locals():
            storage_manager.stop_writer()

    logger.info("aspipe_v4 App4 execution completed")
    return 0
```

## 注意事项

1. 保留了系统原有的 `RateLimiter` 机制，避免双重节流
2. 并发数从8降到6，适度降低而非过度限制
3. 重试机制只在检测到频率限制错误时触发
4. 建议先在小范围数据上测试修改效果
