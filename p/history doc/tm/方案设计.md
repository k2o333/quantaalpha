# StockListManager 方案设计

## 问题背景

当前 `/home/quan/testdata/aspipe_v4/app` 项目中 `stock_basic` 数据存在严重重复下载问题：

### 现状分析
1. **重复调用严重**：在 `date_range_downloader.py` 中发现至少 8 处独立调用 `download_stock_basic()`
2. **无缓存机制**：静态数据没有实现缓存检查，每次都发起 API 请求
3. **资源浪费**：每天可能下载 8-10 次相同数据，浪费 API 调用次数
4. **性能影响**：增加等待时间，可能导致 API 限流

### 具体调用位置
- 第356行：财务数据下载时获取股票列表
- 第463行：股东数据下载时获取股票列表
- 第651行：stk_rewards 数据下载
- 第661行：stk_managers 数据下载
- 第700行：dividend 数据下载
- 第712行：forecast 数据下载
- 第724行：express 数据下载
- tushare_api.py 第303行：cyq_chips 分页下载时获取股票列表

## 解决方案设计

### 1. 核心思想
**程序启动时一次性加载，所有模块共享实例**

### 2. 架构设计

#### 2.1 StockListManager 类
- **单例模式**：确保全局只有一个实例
- **缓存优先**：优先使用本地缓存数据
- **按需下载**：缓存失效时才发起 API 请求
- **统一接口**：为所有模块提供一致的获取方法

#### 2.2 缓存管理器
- **文件格式**：使用 parquet 格式存储（高效压缩）
- **位置管理**：统一管理缓存文件路径
- **有效期检查**：支持自定义缓存有效期（默认24小时）
- **完整性验证**：检查缓存文件的有效性和完整性

#### 2.3 数据新鲜度策略
- **时间驱动**：基于文件修改时间判断是否过期
- **手动刷新**：支持强制刷新缓存
- **失败处理**：下载失败时保留旧缓存数据

### 3. 实现策略

#### 3.1 程序启动流程
```
程序启动 → 初始化 StockListManager → 检查缓存 
    ↓
缓存有效？→ 是 → 加载缓存 → 完成
    ↓
      否 → 下载新数据 → 保存缓存 → 完成
```

#### 3.2 模块调用方式
```
各下载模块 → StockListManager.get_stock_basic() → 返回数据
```

#### 3.3 缓存策略
- **缓存位置**：`cache/stock_basic.parquet`
- **有效期**：24小时（可配置）
- **格式**：parquet（压缩高效）
- **备份机制**：下载失败时保留旧缓存

### 4. 集成方案

#### 4.1 初始化位置
在程序主入口（如 `main.py` 或 `enhanced_main_downloader.py`）中初始化：
```python
# 程序启动时
stock_manager = StockListManager(downloader=tushare_downloader)
```

#### 4.2 现有代码改造
将所有直接调用：
```python
stock_df = self.downloader.download_stock_basic()
```

替换为：
```python
stock_df = StockListManager().get_stock_basic()
```

#### 4.3 逐步迁移策略
1. **第一步**：创建 StockListManager 类
2. **第二步**：在主程序中初始化
3. **第三步**：逐个替换调用点（优先高频调用）
4. **第四步**：清理原有重复代码

### 5. 预期效果

#### 5.1 性能提升
- **API 调用次数**：从每天 8-10 次降低到最多 1 次
- **启动时间**：缓存命中时从几秒降低到毫秒级
- **内存使用**：共享实例减少内存占用

#### 5.2 稳定性提升
- **API 限流风险**：显著降低
- **数据一致性**：所有模块使用同一份数据
- **错误处理**：统一的异常处理机制

#### 5.3 维护性提升
- **代码复用**：消除重复代码
- **配置统一**：缓存策略集中管理
- **日志完善**：统一的日志记录

### 6. 风险评估与应对

#### 6.1 主要风险
1. **单例风险**：单例模式可能导致全局状态污染
2. **缓存风险**：缓存文件损坏可能导致启动失败
3. **并发风险**：多线程环境下可能存在竞态条件

#### 6.2 应对策略
1. **状态隔离**：提供数据副本避免意外修改
2. **容错机制**：缓存失效时自动降级到重新下载
3. **线程安全**：使用锁机制保证线程安全

## 总结

这个方案通过引入 StockListManager 单例模式，配合缓存机制，可以从根本上解决 stock_basic 数据重复下载的问题。预期将 API 调用次数从每天 8-10 次降低到最多 1 次，显著提升系统性能和稳定性，同时保持代码的简洁性和可维护性。
