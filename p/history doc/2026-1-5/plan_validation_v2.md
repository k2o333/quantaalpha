## 2. 新引入的严重缺陷 (必须修复) ❌

### 存储追加策略失效 (Parquet Binary Append)
*   **问题描述**: 方案在 `core/storage.py` 中建议使用二进制追加模式 (`ab`) 来写入 Parquet 文件：
    ```python
    with open(file_path, 'ab') as f:
        df.write_parquet(f, ...)
    ```
*   **技术原理**: Parquet 是一种列式存储格式，文件末尾包含 **Footer (元数据和页脚)**。简单的二进制拼接会将第二个文件的 Footer 放在最后，而第一个文件的 Footer 被埋在中间。
*   **后果**: 
    *   标准 Parquet 读取器（如 `polars.read_parquet`）通常只读取文件末尾的 Footer。
    *   由于第二个 Footer 记录的 Row Group 偏移量（Offset）通常是相对于其自身开头的，拼接后这些偏移量在整个合并文件中将指向错误的位置（指向了第一个文件的数据区）。
    *   **结果**: 文件将**损坏**，或者读取器**只能看到最后一次追加的数据**，之前的数据将“丢失”。

## 修正建议

请修改方案中的 **"2. 存储性能瓶颈修复"** 部分，选择以下任一替代方案：

### 方案 A：Parquet Dataset (推荐 - 性能最好)
*   **原理**: 不再将接口数据存为单个文件 `daily.parquet`，而是存为目录 `data/daily/`。
*   **操作**: 每次写入时，生成一个新的文件名（如 `part-{timestamp}-{uuid}.parquet`）放入该目录。
*   **读取**: `pl.read_parquet("data/daily/")` 支持自动读取目录下的所有分片文件。
*   **优点**: 真正的 O(1) 写入性能，无须读取旧数据，天然支持并发写入。

### 方案 B：Read-Write 重写 (即使有损性能也比损坏好)
*   **原理**: 保持 "读取 -> 合并 -> 去重 -> 重写" 的完整流程。
*   **操作**: 虽然性能较差（方案原意想避免的），但这是维持 "单文件" 结构的唯一安全方式。
*   **优化**: 可以通过设置较大的 `row_group_size` 来稍微优化读取性能，但重写的 IO 开销无法避免。

**结论**: 该方案在存储实现上存在**致命缺陷**，**不可直接执行**。建议采用 **方案 A (Dataset 目录模式)** 进行修正。
