## 2. 核心检测逻辑优化

利用 `app4` 现有的 `StorageManager.read_interface_data` 能力，通过**列过滤**极大降低 I/O 成本。

### A. 窗口级日期覆盖检测 (针对 `date_range`)
不再检测整个任务范围，而是在 `_execute_date_range_pagination` 的每个 `window` 循环中进行检测。

```python
def _is_window_covered(self, interface_name: str, start_date: str, end_date: str) -> bool:
    if not self.storage_manager:
        return False

    # 获取接口配置以确定日期列名
    interface_config = self.config_loader.get_interface_config(interface_name)
    date_col = 'trade_date' # 默认为 trade_date
    # 可以在配置中指定检测用的日期列
    
    # 仅读取日期列，利用 StorageManager 的文件名过滤功能
    try:
        df = self.storage_manager.read_interface_data(
            interface_name, 
            start_date=start_date, 
            end_date=end_date,
            columns=[date_col]
        )
    except Exception:
        return False
    
    if df.is_empty():
        return False

    # 获取该窗口内的预期交易日数量
    trade_cal = self._get_trade_calendar_from_data_dir(start_date, end_date)
    if not trade_cal:
        return False
        
    expected_count = len([d for d in trade_cal if d.get('is_open') == 1])
    actual_count = df.select(pl.col(date_col)).n_unique()

    # 覆盖率阈值（建议 95% 以上视为已覆盖）
    return (actual_count / expected_count) >= 0.95 if expected_count > 0 else False
```

## 4. 关键技术细节与性能优化

1.  **文件名预过滤**：`StorageManager.read_interface_data` 已经通过文件名中的日期信息过滤了文件，检测过程不会扫描全量数据。
2.  **列投影 (Column Projection)**：在调用 `read_interface_data` 时必须显式指定 `columns`，避免 `polars` 加载庞大的财务或行情字段。
3.  **内存缓存检测结果**：对同一个接口在单次运行中的检测结果进行简易缓存，防止同一进程内重复扫描。
4.  **原子性保证**：由于 `StorageManager` 采用 Dataset 模式（多文件存储），即使检测到“已覆盖”并跳过，也不会影响后续新数据的写入。

---

## 5. 预期收益

*   **API 积分节省**：对于已下载过的历史数据（如 `stk_factor_pro` 等高频、大数据量接口），二次运行时的 API 调用量将降低 90% 以上。
*   **存储压力减轻**：不再产生内容完全相同的冗余 Parquet 文件。
*   **系统鲁棒性**：即使下载中断，下次运行也能从中断的窗口附近平滑恢复，而不必从头开始。

---

## 6. 风险点分析

*   **脏数据干扰**：如果现有文件中包含错误的日期数据，可能导致覆盖率误判。
*   **方案对策**：检测逻辑中应结合 `StorageManager` 的确定性去重逻辑，只统计符合主键规范的数据。
*   **主键缺失**：部分接口可能没有明确的日期主键，需根据 `interface.yaml` 动态适配检测字段。
