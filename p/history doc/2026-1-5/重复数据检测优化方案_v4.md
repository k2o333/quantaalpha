## 1. 核心设计理念

为了在保持工程复杂度可控的前提下，满足验收文档中关于 `date_range`、`period_range` 以及 `stock_loop` 的全方位检测需求，本方案引入 **"策略模式 + 轻量级索引"** 的设计。

### 核心策略
1.  **Range Strategy (区间覆盖)**: 适用于 `date_range`。利用交易日历计算时间窗口覆盖率。
2.  **Set Strategy (集合匹配)**: 适用于 `stock_loop` 和 `period_range`。利用轻量级内存索引（Set）快速判断 `ts_code` 或 `period` 是否存在。

## 3. 详细实现策略

### 场景 1: 日期范围 (`date_range`) —— 策略: Range Strategy
**验收目标**: 窗口级检测，跳过已存在的月份/年份。

**实现逻辑**:
1.  **检测列推断**: 优先读配置 `duplicate_detection.date_column`，默认为 `trade_date`。
2.  **IO 优化**: 调用 `storage_manager.read_interface_data(columns=[date_col])`。
3.  **判定**: 
    ```python
    coverage = actual_unique_dates / expected_trade_days
    return coverage >= 0.95
    ```

### 场景 2: 报告期 (`period_range`) —— 策略: Set Strategy
**验收目标**: 精确跳过已下载的季度（如 20230331）。

**实现逻辑**:
1.  **检测列**: 固定为 `period` (或配置指定)。
2.  **索引构建 (Lazy Load)**: 
    *   首次检查时，读取该接口所有历史数据的 `period` 列（仅读取这一列，速度很快）。
    *   将所有 `period` 存入内存 `Set`。
3.  **判定**: `return params['period'] in self.known_periods`。

### 场景 3: 股票历史 (`stock_loop`) —— 策略: Set Strategy (Key-Based)
**验收目标**: 跳过已下载的股票代码。

**工程难点**: `stock_loop` 模式下，如果简单读取所有文件的 `ts_code`，随着数据量增加（如 5000 只股票 * 10 年数据），I/O 会变慢。

**优化方案 (v4 特性)**:
1.  **检测列**: `ts_code`。
2.  **文件级元数据 (File-Level Metadata)**:
    *   利用 `StorageManager` 现有的文件名规范。
    *   **改进**: 建议在写入 `stock_loop` 类数据时，将 `ts_code` 包含在文件名中（目前是 `interface_start_end_ts_uuid.parquet`）。
    *   **如果文件名没包含 ts_code**: 退化为读取所有 Parquet 文件的 `ts_code` 列（Column Projection）。对于 5000 个文件，读取 Header/Column 也是秒级的，完全可接受。
3.  **判定**: 
    *   构建 `existing_stocks = set(all_ts_codes)`。
    *   `return current_stock in existing_stocks`。

## 7. 方案优势

*   **完全满足验收文档**: 覆盖了 `stock_loop`（即 `tscode-historical`）的需求。
*   **性能平衡**: 对于 `stock_loop`，虽然首次运行需要读取所有文件的 `ts_code` 列构建索引，但这是**一次性开销**（Oneshot overhead）。构建完 Set 后，后续几千只股票的判断都是 O(1) 的内存操作，整体速度极快。
*   **代码整洁**: 将检测逻辑剥离到 `CoverageManager`，防止 `GenericDownloader` 变得臃肿。
