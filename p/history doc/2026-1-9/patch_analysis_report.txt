## 潜在问题分析

### 1. 线程安全问题

**问题描述**：补丁中的 `_cache_lock` 使用可能导致死锁

**原因**：
- `should_skip` 获取锁后，内部调用 `_check_range_coverage`
- 如果 `_check_range_coverage` 也尝试获取同一个锁，会导致死锁
- 补丁没有明确说明锁的作用域和释放时机

**建议**：
```python
# should_skip: 只检查缓存，不执行检测逻辑
def should_skip(self, interface_name: str, params: Dict[str, Any], strategy: str = 'auto') -> bool:
    cache_key = (interface_name, tuple(sorted(params.items())))
    
    # 只在缓存检查时使用锁
    with self._cache_lock:
        if cache_key in self._coverage_cache:
            return self._coverage_cache[cache_key]
    
    # 检测逻辑在锁外执行
    result = self._execute_detection(...)
    
    # 更新缓存时加锁
    with self._cache_lock:
        self._coverage_cache[cache_key] = result
    
    return result
```

### 2. 性能影响

**问题描述**：添加 `exchange` 过滤可能导致查询失败

**原因**：
- 某些接口的 parquet 文件可能不包含 `exchange` 字段
- 如果过滤条件中引用不存在的列，Polars 会抛出异常

**建议**：
```python
def _get_trade_calendar_from_data_dir(self, start_date, end_date):
    df = pl.read_parquet(dir_path)
    
    # 构建过滤条件
    conditions = [
        (pl.col('cal_date') >= start_date),
        (pl.col('cal_date') <= end_date),
        (pl.col('is_open') == 1)
    ]
    
    # 检查是否包含 exchange 列
    if 'exchange' in df.columns:
        conditions.append((pl.col('exchange') == 'SSE'))
    
    filtered_df = df.filter(pl.all_horizontal(conditions))
```

### 3. 循环依赖风险

**问题描述**：双重交易日历缓存

**原因**：
- `CoverageManager._make_request_to_trade_cal` 通过 `downloader` 请求 trade_cal 数据
- `downloader` 内部也有交易日历缓存逻辑（`_get_trade_calendar_from_data_dir` + API）
- 可能导致：CoverageManager 缓存一份，Downloader 缓存另一份

**当前实现的问题**：
- CoverageManager 调用 `downloader._make_request()` 绕过了 Downloader 的缓存逻辑
- 导致重复查询和缓存不一致

**建议**：
- 方案1：CoverageManager 直接使用 Downloader 的交易日历缓存，不再单独缓存
- 方案2：将交易日历缓存提取到独立的 `TradeCalendarManager`，两个模块共享

### 4. 缓存一致性问题

**问题描述**：多处缓存没有统一的失效机制

**当前存在的缓存**：
- `downloader._memory_cache['trade_cal']` - 交易日历缓存
- `coverage_manager._cache` - 覆盖率缓存（period/stock）
- `coverage_manager._coverage_cache` - 覆盖率检测结果缓存

**问题**：
- 新数据写入后，这些缓存不会自动失效
- 可能导致使用过期的缓存数据
- 影响数据准确性

**建议**：
- 添加统一的缓存管理器，提供手动清除缓存的方法
- 或者基于 TTL（Time-To-Live）的自动失效机制
- 或者在数据写入后触发相关缓存的失效
