# 接口优化方案

## 性能问题诊断

### 当前实现的性能瓶颈

1. **串行处理**：当前的实现对daily_basic以及多个其他接口采用逐日下载处理（如date_range_downloader.py中194-203行所示的daily_basic，以及后续多行如210-219行的moneyflow_dc，222-231行的moneyflow_ths，234-243行的moneyflow_ind_dc，246-255行的moneyflow_mkt_dc，257-266行的moneyflow_cnt_ths，270-279行的moneyflow_ind_ths，282-291行的stk_factor，294-303行的stk_factor_pro，306-315行的cyq_perf，318-327行的cyq_chips等所示），效率低下。这是重要的性能瓶颈，因为某些接口支持日期范围查询或分页功能，但未被充分利用。

2. **速率限制**：每个API调用都受速率限制控制（tushare_api.py中119-138行），每两次调用之间存在等待时间，这极大地减慢了下载速度，特别是在进行大量API调用时。

3. **独立API调用**：对于多个日度数据接口，代码对每个交易日执行独立的API调用，而没有进行批量处理或并行化。特别是daily_basic, moneyflow_dc, moneyflow_ths, moneyflow_ind_dc, moneyflow_mkt_dc, moneyflow_cnt_ths, moneyflow_ind_ths, stk_factor, stk_factor_pro, cyq_perf, cyq_chips等接口都采用了逐日循环调用方式。

4. **无并发下载**：所有不同类型的数据下载任务都是串行执行的，没有充分利用多核CPU并行下载多种数据类型。

5. **无缓存机制**：系统没有检查已下载的数据文件，不管数据是否已经存在都会尝试重新下载，导致不必要的API调用和重复工作。

6. **错误处理效率低**：当下载失败时，系统会进行指数退避重试，这会增加总时间，且没有有效的失败后跳过机制。

7. **内存管理不足**：在处理大量数据下载时，没有有效的内存控制机制，可能导致内存溢出。

8. **未充分利用接口分页能力**：多个接口支持分页功能（如cyq_perf单次最大5000条，cyq_chips单次最大2000条，stk_factor单次最大10000条等），但代码中未使用offset/limit参数进行分页下载，导致单次API调用返回数据量受限。

## 优化方法

### 1. 并行处理

实现使用ThreadPoolExecutor进行并发下载，同时下载不同的数据类型或同一接口的不同日期范围。具体实现包括：
- 在下载不同数据类型时使用线程池并行执行
- 对于同类型数据的分批处理，也应考虑线程池并行
- 使用threading.Lock确保共享资源的访问安全

### 2. 任务分解与调度

根据不同接口的能力采用不同的下载策略：
- 对于支持日期范围查询的接口，直接使用日期范围参数进行批量下载
- 对于支持分页的接口，使用offset/limit参数进行分页下载
- 对于既不支持日期范围也不支持分页的接口（如daily_basic），按日处理或按小批量日期处理
- 将整个日期范围内需要下载的日期列表分成多个批次
- 为每个批次创建独立的下载任务
- 在线程池中并行执行这些下载任务

### 3. 智能缓存机制

- 实现文件存在性检查，避免重复下载已存在的数据
- 基于时间戳或数据标识符验证数据是否最新
- 支持断点续传，对部分下载失败的数据进行增量下载

### 4. 速率限制优化

- 实现更智能的令牌切换机制（tushare_api.py中已有基本实现）
- 优化API调用间隔，通过更精细的令牌管理提高吞吐量
- 对不同API类型使用不同的速率限制策略

### 5. 优化内存使用

- 在处理大量数据时，使用分批处理方式减少内存占用
- 使用pandas和polars的高效数据操作方法
- 及时释放不需要的中间数据

### 6. 改进错误处理

- 实现更高效的重试机制，避免不必要的重试
- 记录失败的下载任务，便于后续处理
- 添加失败任务的快速跳过机制

### 7. 充分利用接口能力

- 对支持日期范围查询的接口，使用start_date/end_date参数进行批量下载
- 对支持分页的接口，使用offset/limit参数进行分页下载
- 根据接口的单次最大返回记录数合理设置分页大小

## 具体可实施的改进

### 1. 批量日期范围下载与并行处理结合

针对不同的接口特性，采用不同的下载策略：

- **支持日期范围查询的接口**：如 `daily`, `moneyflow`, `moneyflow_dc`, `moneyflow_ths`, `moneyflow_ind_dc`, `moneyflow_mkt_dc`, `moneyflow_ind_ths`, `stk_factor_pro`, `cyq_perf`, `cyq_chips` 等，可直接使用日期范围参数进行批量下载
- **支持分页但不支持日期范围查询的接口**：如 `stk_factor` 等，可在单日下载时使用分页功能
- **既不支持日期范围也不支持分页的接口**：如 `daily_basic`, `moneyflow_cnt_ths` 等，需要按日处理或按小批量日期处理

采用按日期范围分批下载的策略：
- 将整个日期范围分成多个小批次（如每批30天）
- 对支持日期范围查询的接口，按批次进行批量下载
- 对支持分页但不支持日期范围查询的接口，在单日下载时使用分页功能
- 对既不支持日期范围也不支持分页的接口，按批次采用并行处理（在批次内并行处理每个日期）

from concurrent.futures import ThreadPoolExecutor, as_completed
import threading

class ParallelDownloader:
    def __init__(self, max_workers=4):
        self.max_workers = max_workers
        self.tushare_downloader = TuShareDownloader()
        self.download_lock = threading.Lock()

    def download_daily_types_batched(self, data_types, start_date, end_date):
        """
        按批次下载不同类型的数据
        """
        results = {}

        def download_single_type(data_type):
            try:
                if data_type in ['daily', 'moneyflow', 'moneyflow_dc', 'moneyflow_ths',
                               'moneyflow_ind_dc', 'moneyflow_mkt_dc', 'moneyflow_ind_ths',
                               'stk_factor_pro', 'cyq_perf', 'cyq_chips']:
                    # 批量下载支持日期范围的接口
                    return self.download_batched_range(data_type, start_date, end_date)
                elif data_type in ['daily_basic', 'moneyflow_cnt_ths', 'stk_factor']:
                    # 对于不支持日期范围的接口，按小批次并行处理
                    return self.download_daily_type_batched_parallel(data_type, start_date, end_date)
                else:
                    # 对于其他类型的数据
                    return self.download_other_data_type(data_type, start_date, end_date)
            except Exception as e:
                self.logger.error(f"下载 {data_type} 时出错: {e}")
                return None

        with ThreadPoolExecutor(max_workers=self.max_workers) as executor:
            # 按数据类型并行下载
            future_to_type = {
                executor.submit(download_single_type, data_type): data_type
                for data_type in data_types
            }

            for future in as_completed(future_to_type):
                data_type = future_to_type[future]
                try:
                    result = future.result()
                    results[data_type] = result
                    self.logger.info(f"✅ {data_type} 下载完成")
                except Exception as e:
                    self.logger.error(f"❌ {data_type} 下载失败: {e}")

        return results

        """
        # 一次性批量下载

            elif data_type == 'moneyflow':
                df = self.tushare_downloader.download_daily_moneyflow_range(start_date, end_date)
            elif data_type in ['moneyflow_dc', 'moneyflow_ths', 'moneyflow_ind_dc',
                             'moneyflow_mkt_dc', 'moneyflow_ind_ths', 'stk_factor_pro',
                             'cyq_perf', 'cyq_chips']:
                # 调用对应的日期范围下载方法
                method_name = f"download_{data_type}_range"
                if hasattr(self.tushare_downloader, method_name):
                    df = getattr(self.tushare_downloader, method_name)(start_date, end_date)
                else:
                    # 回退到通用方法
                    df = self._generic_batch_download(data_type, start_date, end_date)
            else:
                # 其他支持批量下载的接口
                df = self._generic_batch_download(data_type, start_date, end_date)

            return df
        except Exception as e:
            self.logger.error(f"批量下载 {data_type} 失败: {e}")
            raise

    def download_daily_type_batched_parallel(self, data_type, start_date, end_date, batch_size=30):
        """
        对于不支持日期范围的接口，按日期范围分批，并对每批使用并行处理

        trading_days = self.get_trading_days_in_range(start_date, end_date)

        all_data = []

        def process_batch(day_batch):
            batch_data = []
            for trade_date in day_batch:
                # 检查是否已有缓存数据
                cache_path = self.get_cache_path(data_type, trade_date)
                if not self.is_data_cached(cache_path):
                    try:
                        # 根据数据类型调用对应的下载方法
                        if data_type == 'daily_basic':
                            df = self.tushare_downloader.download_daily_basic(trade_date=trade_date)
                        elif data_type == 'moneyflow_cnt_ths':
                            df = self.tushare_downloader.download_moneyflow_cnt_ths(trade_date=trade_date)
                        elif data_type == 'stk_factor':
                            # 对于stk_factor，使用分页下载
                            df = self.tushare_downloader.download_stk_factor_paginated(trade_date=trade_date)
                        elif data_type == 'stk_factor_pro':
                            df = self.tushare_downloader.download_stk_factor_pro(trade_date=trade_date)
                        elif data_type == 'cyq_perf':
                            # 对于cyq_perf，使用分页下载

                            # 对于cyq_chips，使用分页下载

                            self.logger.warning(f"未知的日度数据类型: {data_type}")

                        if not df.empty:
                            df['trade_date'] = trade_date
                            batch_data.append(df)
                            # 保存到缓存
                            self.save_to_cache(df, data_type, trade_date)
                    except Exception as e:

                    df = self.load_from_cache(cache_path)
                    batch_data.append(df)

        # 对所有批次并行处理

            future_to_batch = {
                executor.submit(process_batch, batch): i
                for i, batch in enumerate(batches)
            }

            for future in as_completed(future_to_batch):
                batch_result = future.result()
                all_data.extend(batch_result)

```python
def download_with_progress_tracking(self, download_func, *args, **kwargs):
    """
    带有进度和性能监控的下载函数
    """
    import time
    start_time = time.time()
    self.logger.info(f"开始执行: {download_func.__name__}")

    try:
        result = download_func(*args, **kwargs)
        elapsed = time.time() - start_time
        self.logger.info(f"完成 {download_func.__name__}，耗时: {elapsed:.2f}s")

        # 根据结果类型记录统计信息
        if isinstance(result, pd.DataFrame):
            self.logger.info(f"  - 数据条数: {len(result)}")
        elif isinstance(result, dict):
            self.logger.info(f"  - 返回分区数: {len(result)}")

    except Exception as e:
        elapsed = time.time() - start_time
        self.logger.error(f" {download_func.__name__} 失败，耗时: {elapsed:.2f}s, 错误: {e}")
        raise
```

### 6. 并行下载不同数据类型

```python
def download_all_available_data_parallel(self) -> Dict[str, any]:
    """
    并行下载所有可用数据类型，优化整体下载效率
    """
    results = {}

    self.logger.info("🚀 开始并行下载所有可用数据...")

    # 创建下载任务列表
    download_tasks = self._create_download_task_list()

    # 根据数据类型优先级分组，按优先级串行，同级并行
    high_priority_tasks = [(name, func) for (name, func, _) in download_tasks
                           if name in ['daily', 'daily_basic', 'moneyflow']]
    medium_priority_tasks = [(name, func) for (name, func, _) in download_tasks
                             if name in ['income', 'balancesheet', 'cashflow', 'fina_indicator']]
    low_priority_tasks = [(name, func) for (name, func, _) in download_tasks
                          if name not in [task[0] for task in high_priority_tasks + medium_priority_tasks]]

    # 并行下载高优先级类型
    self.logger.info("Downloading high-priority data types in parallel...")
    results.update(self._parallel_download_tasks(high_priority_tasks, max_workers=6))

    # 并行下载中等优先级类型
    self.logger.info("Downloading medium-priority data types in parallel...")
    results.update(self._parallel_download_tasks(medium_priority_tasks, max_workers=4))

    # 并行下载低优先级类型
    self.logger.info("Downloading low-priority data types in parallel...")
    results.update(self._parallel_download_tasks(low_priority_tasks, max_workers=3))

    self.logger.info("✅ 所有数据下载完成")
    return results

def _parallel_download_tasks(self, tasks, max_workers=4):
    """
    并行处理给定的任务列表
    """
    from concurrent.futures import ThreadPoolExecutor, as_completed

    results = {}
    failed_tasks = []

    with ThreadPoolExecutor(max_workers=max_workers) as executor:
        # 提交所有任务
        future_to_name = {
            executor.submit(self.download_with_progress_tracking, func): name
            for name, func in tasks
        }

        # 收集所有结果
        for future in as_completed(future_to_name):
            name = future_to_name[future]
            try:
                result = future.result()
                results[name] = result
            except Exception as e:
                self.logger.error(f"任务 {name} 执行失败: {e}")
                failed_tasks.append((name, str(e)))

    if failed_tasks:
        self.logger.warning(f"有 {len(failed_tasks)} 个任务执行失败: {failed_tasks}")

    return results
```

### 7. 增加分页下载支持

在`tushare_api.py`中增加分页下载方法：

```python
def download_with_pagination(self, api_func, limit_per_call=2000, **base_kwargs):
    """
    分页下载数据的通用函数

    Args:
        api_func: API调用函数
        limit_per_call: 每次调用的最大记录数
        **base_kwargs: 传递给API函数的基础参数

    Returns:
        pd.DataFrame: 合并后的所有数据
    """
    all_data = []
    offset = 0

    while True:
        # 添加分页参数
        kwargs = base_kwargs.copy()
        kwargs['offset'] = offset
        kwargs['limit'] = limit_per_call

        try:
            data = api_func(**kwargs)
        except Exception as e:
            self.logger.error(f"分页下载失败, offset={offset}: {e}")
            break

        if data is None or len(data) == 0:
            break

        # 将DataFrame添加到列表中，而不是扩展DataFrame
        all_data.append(data)

        # 如果返回数据少于限制数量，说明已到最后一页
        if len(data) < limit_per_call:
            break

        offset += limit_per_call

    # 将所有数据合并成一个DataFrame
    if all_data:
        return pd.concat(all_data, ignore_index=True)
    else:
        return pd.DataFrame()
```

**内存中积累数据后保存的策略**：

- 对于每个数据类型，每处理完一个批次（如30天）就按日期分组保存文件

### 实际实现总结

根据项目代码，合理的补充说明如下：

- **缓存机制**：当前系统没有实现缓存检查逻辑，每次运行都会重新下载数据。需要按接口优化方案补充缓存检查功能，即在下载前检查对应日期的parquet文件是否存在。
- **内存管理**：已实现分批处理（30天为一批），每批处理完后立即保存。内存中的数据不会无限积累，而是随批次处理随保存。
- **保存策略**：按日期分组保存，每批（最多30个交易日）的数据被分组到对应的年月子目录中。

这种设计既保证了内存使用可控（不会无限积累数据），又提供了分批处理的效率提升。但缺少缓存检查机制，导致重复下载已存在的数据。

**建议补充**：实现缓存检查逻辑，在按批次处理前，先检查该批次中每个交易日的数据是否已存在，对已存在的数据跳过下载，只下载缺失的数据。
