# 重复数据检测优化方案

**项目**: aspipe_v4/app4
**日期**: 2026-01-06
**版本**: v1.0

---

## 问题描述

当前重构后的系统在数据写入前没有检测重复数据，导致相同的数据范围被多次下载和写入不同的文件中。虽然读取时会进行去重，但会造成：
1. 重复下载消耗API积分和网络带宽
2. 生成冗余的parquet文件，增加存储空间占用
3. 降低整体系统效率

---

## 优化目标

实现一个通用的重复数据检测机制，适用于所有接口类型，包括：
- 基于日期范围的接口（如daily, income等）
- 基于报告期的接口（如income_vip, balancesheet_vip等）
- 股票循环接口（如top10_holders, pledge_detail等）
- 静态数据接口（如stock_basic, trade_cal等）

---

## 优化方案

### 1. 写入前重复检测机制

在`GenericDownloader`中实现写入前的重复数据检测逻辑：

#### A. 基于日期范围的检测
对于支持日期范围的接口（pagination mode: `date_range`, `period_range`）：

```python
def _check_existing_data_range(self, interface_name: str, start_date: str, end_date: str):
    """
    检查指定日期范围的数据是否已存在
    返回覆盖率百分比（0-100），100表示完全覆盖
    """
    storage_manager = self.get_storage_manager()  # 通过某种方式获取storage_manager实例
    df = storage_manager.read_interface_data(interface_name, start_date, end_date)

    if df.is_empty():
        return 0  # 未覆盖

    # 获取此时间段内的交易日历
    trade_calendar = self._get_trade_calendar_from_data_dir(start_date, end_date)
    if not trade_calendar:
        return 0

    expected_dates = {day['cal_date'] for day in trade_calendar if day.get('is_open', 0) == 1}
    actual_dates = set(df['trade_date'].to_list()) if 'trade_date' in df.columns else set()

    if not expected_dates:
        return 0

    coverage = len(actual_dates & expected_dates) / len(expected_dates) * 100
    return coverage
```

#### B. 基于报告期的检测
对于报告期范围接口（pagination mode: `period_range`）：

```python
def _check_existing_period_data(self, interface_name: str, periods: List[str]):
    """
    检查指定报告期的数据是否已存在
    """
    storage_manager = self.get_storage_manager()
    covered_periods = set()

    for period in periods:
        df = storage_manager.read_interface_data(
            interface_name,
            columns=['period'] if 'period' in df.columns else ['end_date']
        )

        if 'period' in df.columns:
            existing_periods = set(df['period'].to_list())
        elif 'end_date' in df.columns:
            existing_periods = set(df['end_date'].to_list())
        else:
            continue  # 无法检测，继续

        covered_periods.update(existing_periods & set([period]))

    return len(covered_periods) / len(periods) * 100  # 覆盖率
```

#### C. 基于股票代码的检测
对于股票循环接口（pagination mode: `stock_loop`）：

```python
def _check_existing_stock_data(self, interface_name: str, stock_list: List[Dict], start_date: str, end_date: str):
    """
    检查特定股票在日期范围内的数据是否已存在
    """
    storage_manager = self.get_storage_manager()
    covered_stocks = set()

    for stock in stock_list:
        stock_code = stock['ts_code']
        df = storage_manager.read_interface_data(
            interface_name,
            start_date,
            end_date,
            columns=['ts_code', 'trade_date']
        )

        if not df.is_empty() and 'ts_code' in df.columns:
            stock_data = df.filter(pl.col('ts_code') == stock_code)
            if not stock_data.is_empty():
                covered_stocks.add(stock_code)

    return len(covered_stocks) / len(stock_list) * 100  # 覆盖率
```

### 2. 参数匹配策略

不同接口类型的覆盖检测策略：

| 接口类型 | 参数检测 | 策略 |
|---------|---------|------|
| date_range | start_date, end_date | 检查日期范围覆盖度 |
| period_range | period | 检查具体报告期是否存在 |
| stock_loop | ts_code, start_date, end_date | 检查特定股票+日期范围 |
| offset | 不检测 | 不进行覆盖检测（数据可能动态变化） |
| none | 不检测 | 不进行覆盖检测 |

### 3. 覆盖阈值设置

```python
class GenericDownloader:
    def __init__(self, config_loader: ConfigLoader):
        # ... 现有代码 ...

        # 覆盖率阈值，超过此值则跳过下载
        self.coverage_thresholds = {
            'date_range': 95,    # 95%覆盖则跳过
            'period_range': 100, # 100%覆盖才跳过（报告期数据精确性要求高）
            'stock_loop': 90,    # 90%覆盖则跳过
        }
```

### 4. 下载决策逻辑

```python
def download(self, interface_name: str, params: Dict[str, Any]) -> Optional[List[Dict[str, Any]]]:
    interface_config = self.config_loader.get_interface_config(interface_name)
    pagination_config = interface_config.get('pagination', {})
    pagination_mode = pagination_config.get('mode', 'offset') if pagination_config.get('enabled', False) else 'none'

    # 检查数据覆盖情况
    coverage = 0
    should_skip_download = False

    if pagination_mode in self.coverage_thresholds:
        threshold = self.coverage_thresholds[pagination_mode]

        if pagination_mode == 'date_range':
            start_date = params.get('start_date')
            end_date = params.get('end_date')
            coverage = self._check_existing_data_range(interface_name, start_date, end_date)

        elif pagination_mode == 'period_range':
            # 生成期间列表并检查
            start_date = params.get('start_date', '20050101')
            end_date = params.get('end_date', datetime.now().strftime('%Y%m%d'))
            target_periods = self._generate_quarter_end_dates(start_date, end_date)  # 目标期间
            coverage = self._check_existing_period_data(interface_name, target_periods)

        elif pagination_mode == 'stock_loop':
            stock_list = self._get_stock_list_from_memory_cache()  # 或从data目录获取
            start_date = params.get('start_date')
            end_date = params.get('end_date')
            if stock_list:
                coverage = self._check_existing_stock_data(interface_name, stock_list, start_date, end_date)

        # 根据覆盖率决定是否跳过下载
        if coverage >= threshold:
            logger.info(f"Data coverage {coverage}% exceeds threshold {threshold}%, skipping download for {interface_name}")
            should_skip_download = True

    if should_skip_download:
        # 直接从现有数据中获取
        return self._get_existing_data_by_params(interface_name, params)

    # 执行正常下载流程
    validated_params = self._validate_parameters(interface_config, params)
    all_data = self._execute_pagination(interface_config, validated_params)

    return all_data
```

### 5. 性能优化建议

1. **缓存覆盖检测结果**：对同一接口的覆盖检测结果进行短期缓存
2. **快速路径检测**：对完全相同的参数直接跳过检测
3. **增量检测**：只检测新增的日期范围或参数组合

### 6. 实现优先级

1. **高优先级**：date_range接口（daily, monthly, weekly等）
2. **中优先级**：period_range接口（income_vip, balancesheet_vip等）
3. **低优先级**：stock_loop接口（历史数据变化可能性较高）

## 预期效果

1. **减少重复下载**：避免已存在数据的重复请求
2. **节省API积分**：减少不必要的API调用
3. **优化存储**：减少冗余parquet文件生成
4. **提升效率**：快速跳过已完成的下载任务

---

## 风险评估

1. **数据完整性风险**：需要确保检测逻辑准确，避免误跳过真实需要下载的数据
2. **性能影响**：覆盖检测会增加读取操作，需优化检测算法
3. **边界情况**：部分下载完成的数据需要特殊处理
