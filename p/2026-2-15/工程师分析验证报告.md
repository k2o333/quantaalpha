## 一、工程师核心论断验证

### 论断 1：cyq_chips 出现 API 参数错误

**工程师原话**：
> cyq_chips 在重构后 API 调用直接报错：`API error for cyq_chips: 必填参数, ts_code`

**验证结果**：✅ **正确**

**证据**（来自 [`cyq_chips_1_output.txt`](p/interface4/output/cyq_chips_1_output.txt:28)）：
```
2026-02-15 22:30:31,745 - core.downloader - ERROR - API error for cyq_chips: 必填参数, ts_code
```

**原因分析**：[`UpdateManager._execute_download()`](app4/update/update_manager.py:400) 方法构建的 `params` 只包含 `start_date` 和 `end_date`，没有传递 `ts_code` 参数：

```python
# update_manager.py L419-423
params = {
    'start_date': date_range.start_date,
    'end_date': date_range.end_date
}
```

而 [`cyq_chips.yaml`](app4/config/interfaces/cyq_chips.yaml:43) 配置中 `ts_code` 是必填参数：
```yaml
ts_code:
  description: 股票代码
  required: true
```

### 论断 2：income_vip 和 top10_holders 的缺口检测被跳过

**工程师原话**：
> `is_time_range_mode()` 要求接口的 `pagination.mode` 必须是 `date_range` 或 `reverse_date_range`，但 `income_vip` 和 `top10_holders` 的配置都是 `mode: stock_loop`

**验证结果**：✅ **正确**

**证据 1**：配置文件确认

| 接口 | 配置文件 | pagination.mode |
|------|----------|-----------------|
| income_vip | [`income_vip.yaml:36`](app4/config/interfaces/income_vip.yaml:36) | `stock_loop` |
| top10_holders | [`top10_holders.yaml:25`](app4/config/interfaces/top10_holders.yaml:25) | `stock_loop` |

**证据 2**：代码逻辑确认

[`coverage_manager.py L611-634`](app4/core/coverage_manager.py:611-634) 中的 `is_time_range_mode()` 方法：

```python
def is_time_range_mode(self, interface_name: str) -> bool:
    mode = pagination_config.get('mode', '')
    if mode not in ['date_range', 'reverse_date_range']:
        return False  # stock_loop 模式会在这里返回 False
```

[`update_manager.py L255-257`](app4/update/update_manager.py:255-257) 中的缺口检测入口：

```python
if (options.gap_detection_enabled and
    self.coverage_manager and
    self.coverage_manager.is_time_range_mode(interface_name)):  # ← stock_loop 模式返回 False
    # 缺口检测逻辑（被跳过）
```

## 四、结论

**工程师的分析是完全正确的。**

重构主要改进了代码架构（使用 `UpdateManager`、生成结构化报告），但**并没有实现"下载层面"的增量去重**。核心问题是：

1. `UpdateManager` 的缺口检测逻辑只对 `date_range` 或 `reverse_date_range` 模式生效
2. `stock_loop` 模式的接口（如 `income_vip`、`top10_holders`）完全跳过了缺口检测
3. `cyq_chips` 出现了新的 API 参数错误（`ts_code` 未传递）

工程师提出的修复方案是合理的：需要在 `UpdateManager` 中增加对 `stock_level_detection` 配置的检查，调用 `CoverageManager.detect_stock_gaps()` 方法实现股票级别的缺口检测。
