## 三、根本原因（P0）：异步写入管线的并发安全问题

### 3.1 数据写入管线架构

```
下载数据 → add_to_buffer() → process_queue → _process_worker线程 → data_queue → _writer_worker线程 → 写入parquet文件
```

整条管线通过两个队列（`process_queue` 和 `data_queue`）串联两个后台线程。问题出在关闭管线时的时序错误。

### 3.2 Bug 1：`_process_worker` 循环条件错误

**文件**: `app4/core/storage.py`  
**位置**: 第 540 行

```python
# 当前代码（有问题）
while self.running:          # ← 问题：用 self.running 控制循环
    try:
        task = self.process_queue.get(timeout=1)
        if task is None:
            break
        # ... 处理数据 ...
```

**对比 `stop_writer()` 的执行顺序**（第 117-145 行）：

```
时间线 →
────────────────────────────────────────────────────────
stop_writer()                     _process_worker线程
    │                                    │
    ├─ self.running = False (行120)      │ ← 此刻 running 变为 False
    │                                    │
    ├─ flush_remaining_data() (行123)    │ ← 把缓冲区数据放入 process_queue
    │   → process_queue.put(data)        │
    │                                    │ while self.running: ← False，退出！
    │                                    │ 数据还在 process_queue 中，无人处理
    │                                    │
    ├─ while not process_queue.empty()   │ ← 死等，因为没有消费者了
    │   → time.sleep(0.1)  (行127-128)  │
    │                                    │
    ├─ process_queue.put(None) (行131)   │ ← 哨兵也无人接收
    │                                    │
    ├─ process_thread.join(60) (行133)   │ ← 线程可能已经退出了
```

**关键问题**：`self.running = False` 发生在 `flush_remaining_data()` 之前，`_process_worker` 在 `get(timeout=1)` 超时返回后检查到 `self.running == False`，直接退出循环。此后 `flush_remaining_data()` 放入的数据无人消费。

### 3.3 Bug 2：`_writer_worker` 的竞态条件

**文件**: `app4/core/storage.py`  
**位置**: 第 159-172 行

```python
# 当前代码（有问题）
if item is None:   # 收到哨兵
    while not self.data_queue.empty():   # ← 竞态条件！
        extra_item = self.data_queue.get_nowait()
        if extra_item is not None:
            batch_data.append(extra_item)
    if batch_data:
        self._write_batch(batch_data)
    break   # 退出循环
```

**竞态场景**：

```
时间线 →
────────────────────────────────────────────────────────
_process_worker              data_queue         _writer_worker
    │                           │                     │
    │  [正在处理数据]            │                     │
    │                           │  ← 取出 None 哨兵   │
    │                           │                     │ empty() == True ✓
    │  → data_queue.put(data)   │                     │
    │                           │ [数据已入队]         │ break → 退出！
    │                           │                     │
    │                           │ [数据永远不会被写入]  │
```

`_writer_worker` 取出哨兵后检查 `data_queue.empty()`，但 `_process_worker` 可能正在处理最后一批数据，处理完才会放入 `data_queue`。因此 `empty()` 返回 True 时 `_writer_worker` 直接退出，后续放入的数据丢失。

### 3.4 日志铁证

来自测试报告的时间戳对比：

**成功案例 —— cyq_chips（12402 条，数据量大）**：
```
11:10:55,186 - Processed and queued 12402 records for cyq_chips
11:10:55,193 - Stopping writer thread...
11:10:55,224 - 使用SchemaManager成功创建DataFrame for cyq_chips，记录数: 12402
11:10:55,239 - Wrote 12402 records to data/cyq_chips/...parquet   ← ✅ 写入成功
11:10:55,242 - Storage threads stopped
```

**失败案例 —— income_vip（2 条，数据量小）**：
```
11:11:20,620 - Processed 2 records for income_vip
11:11:20,621 - Processed and queued 2 records for income_vip
11:11:20,621 - Stopping writer thread...                         ← 同一毫秒！
11:11:20,621 - Storage threads stopped                           ← ❌ 没有 Wrote 日志
```

**结论**：数据量小时，处理极快，`Processed and queued` 和 `Stopping writer thread` 在同一毫秒内发生，`_writer_worker` 来不及写入就被终止了。

## 四、次要问题

### 4.1 P1：Type C 接口日期范围传递不完整

**问题**：`stk_rewards` 等 Type C 接口，用户指定 `--start_date 20240101 --end_date 20240630`，但交易日历查询使用了锚点值 `20050101-20240331`。

**根因**：`params_builder.py` 的 `_build_stock_loop_anchor_params_list` 方法未将用户原始日期传入参数字典，`downloader.py` 在获取交易日历时 fallback 到默认值 `20050101`。

**影响范围**：Type C 接口（`stk_rewards`, `top10_holders` 等）的缺口检测日期范围错误。

### 4.2 P2：`clear_dates_cache` 缓存清除不完整

**文件**: `app4/core/coverage_manager.py` 第 597-609 行

```python
def clear_dates_cache(self, interface_name: str = None):
    with self._existing_dates_lock:
        if interface_name:
            self._existing_dates_cache.pop(interface_name, None)
            # ❌ 缺少清除 self._cache 中 "{interface_name}:{ts_code}:dates" 格式的键
        else:
            self._existing_dates_cache.clear()
            # ❌ 缺少清除 self._cache
```

**影响**：`get_stock_existing_dates()` 使用 `self._cache`（键格式 `{interface}:{ts_code}:dates`），与 `_existing_dates_cache` 是不同的字典。清除缓存时只清了一个，另一个中的旧数据可能导致同进程内多轮下载时返回过期结果。

**当前影响程度**：低。因为每次 `python main.py` 运行都是独立进程，内存缓存不跨进程共享。但如果将来改为长驻进程或在同一进程内多次调用 `detect_stock_gaps`，此 Bug 会暴露。

### 4.3 P3：相对路径风险

**文件**: `app4/config/settings.yaml` 第 3 行

```yaml
storage:
  base_dir: "data"   # 相对路径
```

从不同工作目录运行 `python main.py` vs `python app4/main.py` 会导致数据存储在不同位置。

**当前影响程度**：低（测试都是从 `app4` 目录运行）。但长期来看应修复为绝对路径。

## 五、修复方案

### 5.1 P0 修复：`_process_worker` 循环条件

**文件**: `app4/core/storage.py`  
**修改行**: 第 540 行

```python
# 修复前
while self.running:
    try:
        task = self.process_queue.get(timeout=1)

# 修复后
while True:                      # ← 改为 while True，通过哨兵退出
    try:
        task = self.process_queue.get(timeout=1)
        if task is None:         # 哨兵信号
            # 统计日志...
            break
```

**修改说明**：将循环条件从 `while self.running` 改为 `while True`，**只通过哨兵值 `None` 来退出循环**。这样即使 `self.running = False` 已设置，`_process_worker` 仍会继续消费 `process_queue` 中的数据，直到收到 `None` 哨兵。

### 5.2 P0 修复：`_writer_worker` 竞态条件

**文件**: `app4/core/storage.py`  
**修改行**: 第 159-172 行

```python
# 修复前
if item is None:
    while not self.data_queue.empty():
        try:
            extra_item = self.data_queue.get_nowait()
            ...
        except queue.Empty:
            break
    if batch_data:
        self._write_batch(batch_data)
    break

# 修复后
if item is None:
    # 等待 _process_worker 完成最后的数据处理
    # process_thread 在 stop_writer 中先于 writer_thread join
    # 但为安全起见，额外等待一小段时间
    if self.process_thread and self.process_thread.is_alive():
        self.process_thread.join(timeout=30)

    # 现在可以安全地清空 data_queue
    while not self.data_queue.empty():
        try:
            extra_item = self.data_queue.get_nowait()
            if extra_item is not None:
                batch_data.append(extra_item)
        except queue.Empty:
            break
    if batch_data:
        self._write_batch(batch_data)
    break
```

**修改说明**：在 `_writer_worker` 收到哨兵后，先等待 `_process_worker` 线程结束（确保它已将所有数据放入 `data_queue`），然后再清空 `data_queue` 并写入。

### 5.3 P0 修复：`stop_writer` 执行顺序调整

**文件**: `app4/core/storage.py`  
**修改行**: 第 117-145 行

```python
# 修复后
def stop_writer(self):
    """停止所有线程"""
    if self.running:
        # ✅ 先 flush 数据，再设 running = False
        # 确保 _process_worker 还在运行时就把数据放入队列
        self.flush_remaining_data()

        # 等待处理队列中的数据被消费完毕
        logger.info("Waiting for process queue to empty...")
        deadline = time.time() + 60  # 最多等60秒
        while not self.process_queue.empty() and time.time() < deadline:
            time.sleep(0.1)

        # 现在可以安全地停止处理线程
        self.running = False          # ← 移到 flush 之后
        self.process_queue.put(None)  # 发送哨兵
        if self.process_thread:
            self.process_thread.join(timeout=60)
            if self.process_thread.is_alive():
                logger.warning("Process thread did not stop within timeout")

        # 停止写入线程
        logger.info("Stopping writer thread...")
        self.data_queue.put(None)
        if self.writer_thread:
            self.writer_thread.join(timeout=120)
            if self.writer_thread.is_alive():
                logger.warning("Writer thread did not stop within timeout")

        logger.info("Storage threads stopped")
```

**关键改动**：
1. `flush_remaining_data()` 移到 `self.running = False` 之前
2. 等待 `process_queue` 清空的逻辑加了超时保护
3. `self.running = False` 放在哨兵发送之前（但此时 flush 已完成）

def clear_dates_cache(self, interface_name: str = None):
    """清除日期缓存"""
    with self._existing_dates_lock:
        if interface_name:
            self._existing_dates_cache.pop(interface_name, None)

## 八、总结

**一句话诊断**：增量下载去重失效的根本原因是 `storage.py` 的异步写入管线存在并发安全问题——`stop_writer()` 先设置 `self.running = False` 再 flush 数据，导致 `_process_worker` 提前退出、数据未写入磁盘。第二次运行时 `get_stock_existing_dates()` 读取的是空目录，因此认为没有已有数据，触发全量下载。

**修复核心**：将 `_process_worker` 的退出机制从 `while self.running` 改为哨兵模式（`while True` + `None` 退出），并调整 `stop_writer()` 的执行顺序为"先 flush → 再设 running=False → 发哨兵"。
